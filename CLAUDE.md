## Commands

```bash
npm run build       # gen-version + vite build (ES, multi-entry - core + one chunk per plugin subpath) + node scripts/build-umd.mjs (one self-contained UMD file per plugin, own global each) -> dist/ (es + umd + type declarations)

npm run sandbox     # vite dev server for index.html, importing /src/index.ts directly (TS transpiled on the fly) - an alternative to the Live Server workflow below
npm test            # vitest run (jsdom environment)

npm run typecheck   # tsc --noEmit
```

`npm run gen-version` regenerates `src/version.ts` from `package.json`'s `version` field ã do not hand-edit `src/version.ts`.

`index.html` (project root) is a live sandbox exercising every Phase 1 piece against the real `six` API. It imports `./dist/index.js` (the **built** core chunk), specifically so it can be opened with a plain static file server ã VS Code's "Live Server" extension is the intended workflow here, not `npm run sandbox`. Demo pages that exercise a plugin import it separately from its own chunk (e.g. `./dist/Burst.js`, `./dist/SmoothScroll.js`) ã see "Public API" below for the full subpath list. **Run `npm run build` (one-shot) or `npm run dev` (rebuilds on every save) before/while using Live Server**, or the sandbox is testing stale code. MIME type of "video/mp2t"` under strict module-script MIME checking. Verified working via a plain static server (`serve`) with zero console/request errors after switching to the `dist/` import.

## Status: Phase 1 rewrite in progress

This codebase was rebuilt from scratch (see git history starting at "Rebuild Phase 1 core foundation") after an earlier attempt was abandoned for architectural problems ã specifically: lifecycle state lived in a wrapper external to the thing being played (so nested timelines lost repeat/pause/reverse), time was accumulated incrementally per-instance instead of derived by formula (fragile under seeking/timeScale), and the same feature (stagger) got reimplemented three different ways. The rewrite's core design decision is one shared `Animation` base class (`core/animation.ts`) for both `Tween` and `Timeline`, with time flowing as a pure coordinate-transform down the tree rather than being accumulated anywhere except the `Ticker`'s own clock. The full reasoning is in the commit messages (each Phase 1 module landed as its own commit, several documenting a real bug the accompanying tests caught); the short version is inlined as comments at the relevant call sites below.

**A real lesson from this rewrite, worth not repeating**: several early OnScroll bugs (sticky/pin snapping to the viewport top instead of its natural position, the default toggle behavior calling `.reverse()` on leaveBack) came from inventing a plausible-sounding default instead of checking GSAP's actual documented/known behavior first (this plugin was built as, and is still architecturally, a GSAP ScrollTrigger port - it was later renamed to `OnScroll`/`on-scroll.ts` to read as six-js's own, matching the `Breakpoint`/`SmoothScroll` naming precedent below, but the GSAP-parity verification discipline still applies). For anything GSAP-equivalent - a default value, an edge-case behavior, what a specific callback should and shouldn't fire for - verify against GSAP's real behavior before implementing, rather than designing what "seems reasonable." An explicit audit pass cross-checked the engine's remaining defaults against known GSAP `ScrollTrigger` behavior (default `start`/`end`, scrub/`sync` semantics, `markers` default, string `scroller` support, ease still applying under sync, direction-based onEnter/onEnterBack, inclusive range boundaries) and found them already correct; the only other gap found was cosmetic (markers had no visible "start"/"end" text, since fixed).

**A second audit pass covered Core/Tween/Timeline/Animate** (not just ScrollTrigger) and found six real, verified-in-a-real-browser-or-by-test bugs, none of which needed any GSAP-behavior guessing - they were internal consistency bugs, provable from the code's own stated invariants:

1. **`pause()` then `play()` later jumped the animation ahead by however long it was paused, instead of resuming from where it froze.** Root cause: a paused child is simply skipped during its parent's render pass (so its `_tTime` stays frozen - correct so far), but its `_start` (position within the parent's coordinate space) was never adjusted. Once resumed, the very next render recomputed `(parentLocalTime - _start) * timeScale` using the parent's now-much-later local time against the same old fixed `_start` - i.e. it computed "where would this be if it had been playing the whole time," not "where it actually was." This is not a cosmetic issue: it breaks pause/resume for **any** animation attached to the always-ticking root timeline, which includes every top-level `six.to()`/`six.timeline()` AND, critically, every non-sync `OnScroll` animation (`vars.animation.pause()` runs immediately at creation; `.play()` fires whenever the user later scrolls into range) - unfixed, a scroll-triggered reveal would have snapped straight to its end state the instant it was triggered, with no visible animation at all, on any page that had been open for longer than the tween's own duration before the trigger fired. Fixed via a new `AnimationParent._childResumed(child)` hook (`core/animation.ts`) that `Timeline` implements (`timeline.ts`): on the paused->playing transition, it re-anchors `child.startTime()` so the timeline's current local time maps back to exactly the child's frozen totalTime. Skipped only when the timeline's playhead hasn't reached the child's *originally scheduled* start yet (a child still waiting its turn in a sequenced timeline) - deliberately **not** gated on "has the child's totalTime already progressed past 0," which gets the single most common real case (a `paused: true` tween created and resumed later, per above) backwards, since such a tween's frozen totalTime is exactly 0 right up until the moment it's finally played. Verified with a jsdom unit test reproducing the exact scenario (`timeline.test.ts`) AND an end-to-end real-browser Playwright check (a tween created `paused: true`, left alone on the real ticker for 1.5 real seconds, then `.play()`d - confirmed it plays smoothly from 0% rather than jumping to 100%).
2. **`restart(includeDelay)` always replayed the delay, regardless of the flag** - `includeDelay: false` (the default, matching GSAP) is supposed to *skip* the delay on replay; only `includeDelay: true` should wait through it again. The code passed `totalTime(includeDelay ? -delay : 0, true)` - but `render()` unconditionally clamps negative totalTime to 0, so `-delay` and `0` land on the exact same value, making the two modes indistinguishable, and both had the *wrong-for-the-default-case* behavior (0 is the very start of this engine's delay window, since totalTime `[0, delay)` represents "waiting" on this animation's own axis - see `render()`'s doc comment). Fixed to `totalTime(includeDelay ? 0 : this._delay, true)` - skip-delay jumps straight to `_delay` (the boundary where the active portion begins), replay-delay goes to the true `0`. Covered by two new tests in `animation.test.ts`.
3. **A tween ending with a nonzero `z` (or having ever touched `rotationX`/`rotationY`) visibly snapped away its z-offset the instant it settled at rest.** `transform-cache.ts`'s `composeTransform()` chose between `translate3d(x,y,z)` and plain `translate(x,y)` based solely on `use3D` (true only while a tween is actively mid-flight, for GPU-compositing reasons) - at rest, `use3D` is false, so it always rendered `translate(x,y)`, which has no `z` component at all, silently discarding it. Fixed so a nonzero `z` always forces `translate3d`, regardless of `use3D` - `use3D` now only controls the compositing-layer promotion for the common `z === 0` case, never a license to lose real state. Covered by two new tests in `transform-cache.test.ts`.
4. **`restart()` on an animation that had already finished playing naturally (never explicitly paused) left it snapping straight back to its completed/end state instead of actually replaying.** `_childResumed` re-anchoring (see bug 1) only fires from `paused()`'s paused->playing transition (`resuming = value === false && this._ts === 0`); a tween/timeline that reached the end of its own duration without ever being paused still has `_ts !== 0` the whole time, so `restart()`'s `totalTime()` reset was silently overwritten on the very next render by the parent (e.g. the always-ticking root) recomputing `(parentLocalTime - _start) * timeScale` against the same stale, long-past `_start` - clamping straight back to "complete" rather than 0. This broke the ordinary "replay" use case for *any* top-level `six.to()`/`six.timeline()` once played to completion, found while building a Timeline feature-coverage demo (`public/timeline-demo.html`) where repeated Play-button clicks stopped animating after the first. Also revealed the same gap applied to `.reverse()`: changing `timeScale()`'s sign on a live (non-paused) child never re-anchored `_start` either, so reversing something that had been running for a while didn't animate backward at all - it clamped straight to 0 on the very next frame. And `play()`/`resume()` had been collapsed into one identical method (`{ this.paused(false); }`), so there was no way to force forward playback after a reverse - contradicts GSAP's actual, distinct semantics (`play()` always plays forward; `resume()` and `reverse()` preserve whatever direction is set). Fixed in `core/animation.ts`: `timeScale()` now calls `this.parent?._childResumed?.(this)` unconditionally on every change (reused by `restart()` via `play()`, and by `.reverse()`/`.timeScale()` directly), `play()` now calls `reversed(false)` before unpausing, and `resume()`/`reverse()` call `paused(false)` directly instead of delegating to `play()`. Covered by new tests in `animation.test.ts` and `timeline.test.ts`.
5. **A child paused inside a *bounded* nested timeline could end up permanently frozen on resume if enough real time passed first, even though the exact same pause/resume worked fine when done quickly.** `_childResumed`'s re-anchor math (bug 1) used `this._lastRenderedLocal` - the parent timeline's own last-rendered local time - as "now." That cached value goes stale the moment the parent itself stops being actively rendered, which is exactly what happens to a bounded nested timeline once its own span is exceeded: `Timeline._renderIteration`'s skip-range optimization (`if (end < lo || start > hi) continue`) means its *own* parent stops calling `render()` on it entirely once real time passes its end, even while it's still logically "not paused." Once that happens, `_lastRenderedLocal` freezes; a `_childResumed` reanchor computed off it afterward understates how much real time has actually passed, producing a startTime that still falls short of "now" - which usually means the child's own newly-extended span *still* doesn't reach the current time, so the parent keeps being skipped forever and the child is stuck at whatever position it was paused at, permanently. Confirmed both broken and fixed by actually running reproductions through the real public API (`six.timeline()` + `rootTimeline`) via `vitest`, not by inspection alone - a stale-value trace like this is easy to get wrong by hand. Fixed with a `currentLocalTimeOf()` helper (`timeline/timeline.ts`) that computes a timeline's true current local time by walking up through parents via each ancestor's own current `totalTime()` (bottoming out at the ultimate root, which is never skipped and so is always accurate) instead of trusting any single cached `_lastRenderedLocal`. `_childResumed` now uses this instead of `this._lastRenderedLocal`. Covered by a new test in `timeline.test.ts`.
6. **A zero-duration animation (`.set()`/`.call()`) added "now" onto the always-ticking root could get a redundant second render one real tick after creation, silently reverting any *later*-created sibling's write to the same target back to its own value.** A `Tween`'s constructor already self-renders once synchronously (`this.render(0, true, true)`), applying a `.set()`'s values immediately - that's correct and sufficient, since a zero-duration animation's output is `eased(1)` regardless of *when* it's rendered (never a function of elapsed time). But because the synchronous script that creates it never advances root's own clock, the child's `startTime` (root's "now" at creation) ends up exactly equal to root's `_lastRenderedLocal` from the *previous* real tick - and `Timeline._renderIteration`'s boundary-inclusive skip check (`end < lo`, not `<=`) treats that as "not yet rendered," giving it one more real render through the parent on the very next tick. That second render is completely redundant on its own - but if a sibling created *after* it in the same script (e.g. a `six.to(..., { onScroll })` reveal that gets synchronously jumped to its complete state - see `OnScroll`'s "already past on construction" fix below) already wrote a *different* value to the same target before that tick fires, the zero-duration child's redundant render silently stomps it back. Found while debugging a report that a scroll-reveal's first few words never appeared - traced through real browser console output (`getComputedStyle` showing the pre-set random `y`/`rotate` values, not the revealed ones, even though the tween's own `totalProgress()` correctly read `1`) down to a `ticker.tick()`-driven reproduction, not by inspection alone. Fixed in `Timeline._renderIteration`: additionally skip a child when `child.totalDuration() === 0 && start <= lo` - a zero-duration child only still needs its one real render when its scheduled position is *strictly ahead* of the low-water-mark (genuinely not yet reached), not when it merely coincides with it. Covered by two new tests in `timeline.test.ts` (one confirming the redundant render is now skipped, one confirming a genuinely future-scheduled zero-duration child still renders once it's actually reached).

**`.reverse()`ing an infinite-repeat (`repeat: -1`) animation that had ever been paused/resumed (or otherwise re-anchored) permanently corrupted it to `startTime = -Infinity`, breaking every future render and making it unrecoverable even via `.restart()`/`.play()`.** Found live via the `svgMotion` demo (`svg-motion-demo.html`): a `repeat: -1`, `autoRotate`-following plane on a `motionPath` had a "Reverse" button; clicking it once and the plane never moved sanely again, not even after "Restart". Root cause: `Timeline`'s `toChildTotalTime()`/`_childResumed()` (`timeline.ts`) both compute a `reversed`-playback offset as `ts >= 0 ? 0 : tDur` - the idea being that reversed playback starts counting down from the animation's own end (`tDur`) rather than up from 0. For `repeat: -1`, `totalDuration()` is `Infinity` (see `cycle.ts`'s `totalDurationOf`) - there is no "end" to count down from, so using `tDur` directly there feeds `Infinity` straight into the surrounding arithmetic, and `_childResumed`'s re-anchor (`now - (childTotalTime - offset) / ts`) collapses to `-Infinity`. Every later render for that child then computes NaN/nonsense localTime. Fixed with a small shared `reverseOffset(ts, tDur)` helper (replacing both inline `ts >= 0 ? 0 : tDur` expressions) that falls back to `0` whenever `tDur` isn't finite - structurally identical to the forward-playback case, i.e. "keep counting from wherever totalTime already is" - which is the only coherent reading of "reversed" when there's no finite end to reference. Verified broken and fixed via a direct repro (`new Tween(el, { repeat: -1 }).reverse()` under a live `Timeline`) before writing the regression test in `timeline.test.ts`. This is a general core bug, not specific to `svgMotion` - any `six.to()`/`six.timeline()` with `repeat: -1` that gets `.reverse()`d after having been paused/resumed once (which includes anything driven by `OnScroll`, since `vars.animation.pause()` runs at construction) would have hit it.

**`TweenVars.overwrite?: boolean | "auto"` is now implemented** (`tween/overwrite.ts`): a `WeakMap<Element, Map<string, Tween>>` tracks which `Tween` currently "owns" each `(target, prop)` pair. Every tween registers its own claimed pairs on init, regardless of its own `overwrite` setting (so a *later* tween can detect and overwrite it even if *this* one didn't ask for protection) - `overwrite` unset/`false` still results in no kills, matching GSAP's own default. `overwrite: true` kills the *entire* previous owning tween (all its targets/properties, not just the conflicting one) - matches GSAP's documented "regardless of which properties are being animated" behavior. `overwrite: "auto"` instead surgically removes just the conflicting `(target, prop)` track from the owner via `Tween._dropTrack()`, leaving its other properties/targets animating undisturbed, and only kills the owner outright if that was its last remaining track. Required `ColorTrack`/`ComplexTrack`/`DiscreteTrack` to start carrying their `prop` name too (previously only `NumericTrack` did) so `"auto"` can match on it. Keyframe-based tweens are excluded (documented scope cut - `_onInit` only calls `applyOverwrite` in the non-keyframe branch, since a keyframe tween's tracks are built lazily per-segment rather than up front). Covered by tests in `tween/overwrite.test.ts`.

**Core, Tween, Timeline, Animate, OnScroll, SvgMotion, SmoothScroll, Burst, ScrambleText, SplitText, and Parallax are implemented** (each covered in its own Architecture section below), alongside a fully independent set of `sx-*` web components (`src/components/` - marquee/dialog/slider, see the Components section). Draggable, Flip, and Observer (the gesture primitive, not `on-scroll/observer.ts`) remain unported - genuine scope cuts, not yet attempted. GSAP's ScrollSmoother itself is also unported, but `SmoothScroll` (Lenis-modeled, see its own section) covers the same whole-page-smooth-scroll use case via a different reference implementation. Property renaming and API trimming are still deferred **for the core engine's own vars/behavior** - don't rename `TweenVars`/`TimelineVars`/`Animation` fields or change engine semantics opportunistically. That restriction does not extend to the public export *surface*: the tree-shaking restructuring (see "Public API" below - a lean `six` core plus one subpath per plugin) was a deliberate, explicitly-requested exception, not a violation of it.

**`LEGACY_CLEANUP.md`'s claim that `src/components/`, `src/properties/`, and several `src/core/*.ts` files are dead is wrong for most of them** - re-verified directly against the current import graph. `src/components/` (all three `sx-*` components), `src/properties/transform-state.ts`, and `src/core/observer.ts`/`inertia-physics.ts`/`safe-element.ts`/`time.ts` are genuinely live - real, shipped code behind the `@six-js/core/Components` subpath - but **not** transitively pulled in by `src/index.ts`/`api/six.ts` at all (that import was deliberately removed along with `enableElements()`, see the Components/Public API sections above, so the ~4,000-line components tree stays fully out of the core bundle). `tsconfig.json`'s `exclude` list previously relied on that now-removed transitive import to still get `src/components/**` type-checked despite excluding it by name - once the import was gone, `tsc --noEmit` silently stopped looking at `src/components/**` at all (confirmed via `tsc --noEmit --listFiles`: zero `components/` files in the compiled program), a real regression only caught via the VS Code TypeScript language service flagging an error `tsc --noEmit` itself missed. Fixed by removing `"src/components"` from `tsconfig.json`'s `exclude` array - it's back to being discovered directly via the `"src"` include glob, independent of any particular import graph shape. `src/properties/` and the individual `core/observer.ts`/`inertia-physics.ts`/`safe-element.ts`/`time.ts` excludes are left as-is; they're still transitively reached (and so still checked) via `src/components/**`'s own imports of them, the same mechanism that used to cover components itself. The one file in the original legacy list that **is** still genuinely dead is `src/core/breakpoints.ts` (the old media-query helper - distinct from the current, live `src/core/breakpoint.ts`/`Breakpoint` class, see Core below); nothing imports it anymore. `LEGACY_CLEANUP.md` itself still needs a matching rewrite - a real follow-up someone should do, not done here since it wasn't the ask - flagged so it isn't mistaken for still-accurate.

## Architecture

### Core (`src/core/`)

- **`ticker.ts`** ã `Ticker` class, one shared instance (`ticker`) drives everything. rAF-based with lag smoothing (a backgrounded tab doesn't cause a catch-up jump on refocus) and a `{ manual: true }` mode (bypasses rAF/the overlap-gap gate entirely, driven only by `.tick(deltaMs)`) used throughout the test suite for deterministic timing.
- **`animation.ts`** ã `Animation`, the abstract base both `Tween` and `Timeline` extend. Owns the entire time model (`_start/_dur/_tDur/_time/_tTime/_ts/_rts`) and lifecycle (`play/pause/resume/reverse/restart/seek/kill/invalidate`, event emitter). `_ts === 0` *is* the definition of paused; the sign of `_rts` *is* the definition of reversed ã no separate boolean flags to keep in sync. `render(totalTime, suppressEvents, force)` is concrete (not abstract): it does the generic clamping/repeat-cycle/event-firing work and delegates only the "apply visual state for this local time" part to the abstract `_renderIteration()`. **Deliberately does NOT skip re-rendering when `totalTime` looks unchanged** ã see the long comment in `render()`; two different logical render passes (e.g. a sibling touching the same target in between) can compute the identical totalTime for an animation, and skipping on value-equality alone silently drops a needed re-render. A zero-total-duration animation (`.set()`/`.call()`) always clamps to totalTime 0, so before/after boundary detection there uses the *raw* pre-clamp totalTime's sign (`_rawPrev` field) instead of the clamped value. `paused(false)` (resuming) calls `parent?._childResumed?.(this)` ã see `Timeline._childResumed` ã a real, high-impact bug (animations jumping to their end state on resume) lived here; read the Status section above before touching pause/resume. `restart(includeDelay)` totalTime target is `includeDelay ? 0 : this._delay` ã get this backwards and the two modes become indistinguishable (both clamp to the same value), which is exactly the bug that shipped once already.
- **`cycle.ts`** ã `resolveCycle()`, the one shared repeat/yoyo formula (floor-division based) both `Animation.render()` call sites use, instead of GSAP's two independently-duplicated inline copies.
- **`linked-list.ts`** ã generic sorted doubly-linked-list helpers (`insertSorted`/`removeNode`/`forwards`/`backwards`), reused for `Timeline` children.
- **`root.ts`** ã `rootTimeline`, the one `Timeline` registered with `ticker`. Constructed `{ unbounded: true, defaultPosition: "now" }`: `unbounded` means its own `totalDuration()` is always `Infinity` (never clamped by whatever children currently exist ã without this, once ticker time exceeds the current max child span the root's own `_tTime` gets stuck and can never advance again even as new children are added), and `defaultPosition: "now"` means a child added with no explicit position starts at the root's *current* playhead, which is what makes independently-created top-level tweens play immediately/concurrently instead of queuing. Kept always-on for Phase 1 rather than implementing GSAP's autoSleep (rAF-loop teardown when idle) ã documented scope cut, not an oversight.
- **`context.ts`** ã `Context`/`context()` (`six.context()`). A module-level active-scope stack; `Animation`'s constructor calls `getActiveScope()?._capture(this)` unconditionally, so both `Tween` and `Timeline` get auto-capture-on-create for free without `context.ts` knowing either class exists (it only deals with a generic `Killable` interface ã zero dependency direction from context ã animation).
- **`defaults.ts`** ã `getDefaults()`/`setDefaults()` (`six.config()`), just `{ duration, ease }` fallbacks.
- **`unregistered-elements.ts`** ã `watchForUnregisteredElements()`, called unconditionally from `api/six.ts` (so it's always active for every `@six-js/core` consumer, not just ones that import `@six-js/core/Components`). One-time scan, after the DOM is ready, for any `sx-*` tag that isn't a registered custom element - `console.warn`s per distinct missing tag. Deliberately tag-*pattern*-only (no hardcoded list of real component tags, no import from `src/components/`) - that's both what lets it live in core without pulling in any component code, and what lets it catch a page that never imports `@six-js/core/Components` at all, which a check living inside the components themselves structurally cannot (their code simply never runs in that case) - see the Components section below for the real bug this was found from. Queries `:not(:defined)` (native Custom Elements v1 pseudo-class - matches only not-yet-upgraded hyphenated tags, filtered by the browser's own selector engine), **not** `"*"` - a real perf concern this codebase caught before shipping: `"*"` would force a JS-side `tagName` check against every element in the document, an O(page size) cost paid by every consumer regardless of whether they use any component at all, whereas `:not(:defined)` narrows the JS-side work down to genuine candidates (typically zero). Deferred via `requestIdleCallback` (falls back to `setTimeout` where unavailable) rather than `queueMicrotask`, since this is a dev-experience warning that should never compete with real work during the page's own critical startup window.
- **`breakpoint.ts`** ã `Breakpoint`/`six.breakpoint()`, GSAP `gsap.matchMedia()`-equivalent: ties setup/teardown to CSS media query state instead of the caller hand-rolling `change` listeners. Built on `Context` rather than duplicating its capture/revert machinery - each `.add()` gets its own `BreakpointContext` that's `revert()`ed (soft - listeners stay attached, a later matching change runs it again) whenever its condition(s) change, and `kill()`ed (hard - also detaches the underlying `MediaQueryList` listeners) only when the whole `Breakpoint` is killed. Re-fetches a fresh `MediaQueryList` on every check rather than trusting a cached one - Firefox only updates a `MediaQueryList`'s own `.matches` at the moment *its own* change handler fires, so a different query's stored object can read stale otherwise; checked directly against GSAP's own `gsap-core.js` (`MatchMedia`/`_onMediaChange`) rather than guessed, per this file's own verify-against-GSAP discipline. A named-query map (vs a single string) coalesces same-tick toggles via one `queueMicrotask` before re-checking, so a resize that flips two queries "together" (e.g. `isDesktopãfalse` and `isMobileãtrue` from the same event pair) doesn't briefly revert into a halfway state between the two native `change` events. Named `Breakpoint`, not `MatchMedia`/`Media` - deliberately not mirroring GSAP's naming (same "reads as six-js's own" philosophy as `OnScroll`/`SmoothScroll`), and distinct from the unrelated, genuinely-dead `src/core/breakpoints.ts` (old media-query helper) and from `ContainerBreakpoints` in `src/core/container-breakpoints.ts` (the independent `sx-slider`/`sx-dialog` web components' own container-width config parser, unrelated to media queries at all - see Components below).

### Tween (`src/tween/`)

- **`tween.ts`** ã `Tween extends Animation`. Normally builds `PropertyTrack[]` from vars via `animate/registry` lazily on first render (`_onInit()`); when `vars.keyframes` is given it instead composes (has-a, not is-a) an internal `Timeline` of `.fromTo()` segments (`keyframes.ts`) and just drives that from `_renderIteration()`. The constructor always calls `this.render(0, true, true)` immediately so e.g. a `.from()` tween's starting values are visible synchronously, not one tick later. `_onInit()` calls `applyOverwrite()` (`overwrite.ts`) right after building tracks, for non-keyframe tweens only. `_dropTrack(target, prop)` (called by `overwrite.ts`'s `"auto"` mode) surgically removes one track and self-kills once none remain.
- **`overwrite.ts`** ã implements `TweenVars.overwrite`. A `WeakMap<Element, Map<string, Tween>>` tracks which `Tween` currently owns each `(target, prop)` pair; every tween registers its own claimed pairs on init regardless of its *own* `overwrite` setting (so a later tween can still detect and overwrite it, even though *this* one didn't request protection) ã leaving `overwrite` unset costs nothing extra and correctly kills nothing, matching GSAP's own default (`false`). `overwrite: true` kills the *entire* previous owner (all its targets/properties, not just the conflicting one) ã matches GSAP's documented "regardless of which properties are being animated" behavior for `true`. `overwrite: "auto"` instead calls the owner's `_dropTrack()` for just the conflicting `(target, prop)` pair, leaving its other properties/targets undisturbed, and only kills it outright once it has no tracks left.
- **`property-track.ts`** ã the actual interpolation records (`NumericTrack | ColorTrack | ComplexTrack | DiscreteTrack`, all four now carrying their own `prop` name so `overwrite.ts` can match on it) and `buildTracks()`/`applyTrack()`. Transform properties write into the shared per-element cache (`animate/transform-cache.ts`); `Tween._renderIteration()` collects which targets were touched and composes+writes the final `transform` string **once per element per frame**, not once per transform sub-property.
- **`keyframes.ts`** ã `buildKeyframeTimeline()`. Each keyframe becomes a `.fromTo()` segment; a `carry` object explicitly threads each segment's own end values forward as the next segment's start (rather than letting the next segment "read current DOM state", which by the time its own track is built lazily may not yet reflect the previous segment's outcome).

### Timeline (`src/timeline/`)

- **`timeline.ts`** ã `Timeline extends Animation`, implements `AnimationParent` (children call `parent._removeChild(child)`/`parent._uncache()` ã structural interface, so `core/animation.ts` never imports `Timeline`) and `ListHandle<Animation>` (backs the generic linked-list). `_recomputeTotalDuration()` scans children for `max(child.endTime())`. `_renderIteration()` converts parent-local time into each child's own totalTime via `toChildTotalTime()` (`(parentLocalTime - child.startTime()) * ts + (ts >= 0 ? 0 : tDur)`) ã implemented purely through `Animation`'s public API (`startTime()/timeScale()/paused()/totalDuration()`), never protected-field access, so there's no coupling between sibling subclasses. **Only renders children whose `[start, end]` overlaps the span actually crossed since the last render** (`lastLocal..localTime`, in either direction) ã rendering every child unconditionally every frame was tried first and is unsafe whenever two children share a render target with different values (chained keyframe segments, or two `.to()` calls on the same property at different timeline positions): a still-in-the-future child clamps to *its own* start value and, iterating after an active sibling in start-time order, silently overwrites the active one's just-computed value. `Timeline.add()` widens the low-water mark back to a newly-added child's start so a child inserted behind the current playhead is still picked up on the next render. A child with `totalDuration() === 0` (a `.set()`/`.call()`) is additionally skipped once `start <= lo` - see bug 7 in the Status section for why a zero-duration child's render is provably idempotent once its position has already been reached, making a second one not just wasteful but actively harmful if a sibling wrote to the same target in between. `kill()` is overridden to cascade to every child before detaching itself (the base `Animation.kill()` only detaches from its own parent). `_childResumed(child)` implements the `AnimationParent` hook `Animation.paused()` calls on resume (and that `timeScale()` now also calls on every change, and `restart()` via `play()`) ã re-anchors `child.startTime()` using `currentLocalTimeOf(this)` (this timeline's own *true current* local time, computed fresh by walking up through parents rather than trusting the potentially-stale cached `_lastRenderedLocal` ã see bug 5 in the Status section) so the child continues from its frozen totalTime instead of jumping to wherever the unadjusted `(parentLocalTime - _start)` formula now points. Guarded by `now < child.startTime()` (not by whether the child's totalTime had already progressed) ã see the long comment on the method for why the naive "hasn't progressed yet" check gets the most common real case (an `OnScroll`-driven, `paused: true`-from-creation animation resumed later) backwards.
- **`position-parser.ts`** ã pure `resolvePosition()` for `"<"`/`">"`/`"+=1"`/label-reference position strings, independently testable against a small context object (no `Timeline` construction needed).
- **`stagger.ts`** ã `computeStaggerDelay()`. Not a parallel "stagger group" primitive: `Timeline.to(targets, { stagger })` builds one `Tween` per target, each with its own `delay`, all added at the *same* timeline position ã reuses `Timeline`/`Tween` entirely.

### Animate (`src/animate/`)

The property/DOM-rendering layer ã what `Tween` calls to actually read/write the DOM (GSAP's `CSSPlugin` equivalent).

- **`registry.ts`** ã `resolveHandler(target, prop, sampleValue)` dispatches to a `numeric | color | complex | discrete` handler. Resolution ladder: transform alias ã known color/complex prop ã CSS custom property (`--foo`, numeric vs discrete decided from `sampleValue`) ã generic style property (numeric if `sampleValue` looks numeric) ã genuine plain-number non-style property (`scrollTop`-style; **only** when `typeof target[prop] === "number"`, not merely `prop in target` ã an existing-but-non-primitive property like an SVG element's `cx` (`SVGAnimatedLength`) must fall through to `setAttribute` instead of being naively overwritten with a raw number) ã `setAttribute` fallback.
- **`transform-cache.ts`** ã per-element cache (`WeakMap<Element, TransformCache>`, not an expando property ã a deliberate simplification vs GSAP's `target._gsap`, revisit only if profiling says it matters). `parseMatrixString()` (exported standalone, unit-testable without a real browser) decomposes an existing `matrix()` fully (translate/rotate/scale/skew) but only reads translate x/y/z out of an existing `matrix3d()` ã full 3D decomposition is a documented Phase 1 scope cut. `composeTransform(cache, use3D)` prefers `translate3d` while `use3D` is true (the caller passes `ratio > 0 && ratio < 1`, i.e. actively animating), falling back to plain `translate()` at rest so elements aren't left on a GPU compositor layer unnecessarily ã **but a nonzero `z` always forces `translate3d` regardless of `use3D`**. It used to fall back to `translate(x, y)` purely off `use3D`, which silently dropped `z` entirely (that function only takes 2 args) the instant any tween touching `z` came to rest - a real, verified bug, not a hypothetical one; `use3D` is a compositing-performance hint for the default `z === 0` case, never a license to lose real data.
- **`unit-convert.ts`** ã measurement-based unit conversion (rem via root font-size, em via the target's own font-size, vh/vw via viewport size, % via the parent's `clientWidth`/`clientHeight`). The % path needs real layout and so only produces a meaningful result in a browser, not jsdom (documented in the file and in the test suite).
- **`color.ts`** ã parses hex/rgb/rgba (comma and modern space syntax)/named colors. Named-color resolution is a **hardcoded table**, not resolved through the DOM/CSSOM (`element.style.color = name; read back`) ã jsdom's CSSOM doesn't normalize named colors the way real browsers do, which would make parsing silently environment-dependent; the hardcoded table is deterministic everywhere.
- **`complex-string.ts`** ã tokenizes numbers *and* embedded colors out of multi-value CSS strings (`box-shadow`, etc.), interpolates each positionally, reassembles using the end string's literal segments as the template. `canInterpolateComplex()` gates whether `property-track.ts` uses this path at all vs degrading to a discrete swap when start/end have structurally different token shapes.

### OnScroll (`src/plugins/on-scroll/`)

A GSAP `ScrollTrigger`/`Observer.js` port, split the same way GSAP splits them, then renamed to read as six-js's own (`ScrollTrigger` class ã `OnScroll`, `scroll-trigger.ts` ã `on-scroll.ts`, `pin` ã `sticky`, `scrub` ã `sync`, `containerAnimation` ã `syncTo`) - same naming philosophy as `Breakpoint`/`SmoothScroll` below ("the user asked for something that reads as six-js's own, not a renamed port"). The rename was code-and-tests-complete; **this section was not updated to match until now** - if anything elsewhere in this file still says `ScrollTrigger`/`scroll-trigger.ts`/`pin.ts`/`scrub.ts`/`containerAnimation`, it means the same terminology drift, read it as `OnScroll`/`on-scroll.ts`/`sticky.ts`/`sync.ts`/`syncTo`. All GSAP-parity behavior/bug history below is otherwise unchanged by the rename - only names moved, not semantics.

- **`observer.ts`** ã generic scroll/resize listening + a memoized-read layer keyed off a module-level "generation" counter (`invalidateReads()` bumps it; `getScroll()` only performs the real, layout-touching read once per generation no matter how many callers ask). Also the shared read layer `SmoothScroll` reads/writes through (see below) - zero dependency on `on-scroll.ts` itself. Axis-aware (`getScroll(scroller, axis)`/`getViewportSize(scroller, axis)`) so both `OnScroll`'s `axis: "x" | "y"` and `SmoothScroll`'s own `axis` vars share one implementation rather than each hand-rolling horizontal/vertical branching. `addResizeListener()` also re-runs listeners once on `window`'s `load` event if `document.readyState` isn't `"complete"` yet when first called - an `OnScroll`'s first `refresh()` (during construction) can run before images/fonts below the trigger finish loading, so without this its initial start/end measurement is taken against a document that's still going to grow.
- **`sticky.ts`** ã spacer creation is cached **per stuck element** (`WeakMap<HTMLElement, ...>`, reference-counted), not per `OnScroll` instance, so multiple triggers sticking the same element share one spacer. Three explicit phases (`"before" | "during" | "after"`) rather than toggling a single `position: fixed` on/off: `"after"` uses `position: absolute` with `top: distance` *within the spacer* so the element resumes normal document flow once scrolled past, instead of staying artificially fixed. The `"during"` phase's `top` is **not** hardcoded to `0` - it's `naturalDocTop - startY` (`setStickyTop()`, computed in `on-scroll.ts`'s `refresh()`), i.e. wherever the element naturally sat in the viewport at the moment scroll reaches `startY`. Hardcoding `top: 0` was a real shipped bug: it only looked correct for a `start: "top top"` trigger (where that offset genuinely is 0) and silently snapped anything stuck via `"center center"` or similar to the viewport's top edge instead of its natural (e.g. vertically centered) position - and made the element flicker between that wrong position and its normal unstuck one whenever scroll hovered right at the `start` boundary, since `"before"`/`"during"` disagreed about where it should be. `distance` (`endY - startY`) is what keeps the `"during"`ã`"after"` transition seamless regardless of the sticky top's value - this falls out of the geometry algebraically, not from any extra `"after"`-phase adjustment. `setPhase()` no-ops when the phase hasn't actually changed (a fast/fling scroll can fire many native `scroll` events per rendered frame; rewriting the same style values on every one is wasted recalculation, not a visual no-op) - `setStickyTop`/`setDistance` explicitly reset that guard since they're the only things that can change what a given phase should render.
- **`sync.ts`** ã `createDirectSync()` sets progress synchronously 1:1 (`sync: true`). `createSmoothSync(animation, seconds)` (`sync: <number>`) retargets on each scroll event: fetched and verified against GSAP's actual `ScrollTrigger.js` source (`scrubDuration`) rather than inventing a plausible-looking formula - real GSAP builds an internal tween with `ease: "expo"` (i.e. `expo.out`) and `duration: scrub`, then on every scroll event either calls `scrubTween.resetTo("totalProgress", newTarget, ...)` or (fallback) sets `vars.totalProgress` and calls `.invalidate().restart()` - either way, a **fresh `expo.out` ease from the current value to the new target, over exactly the declared seconds**, restarted from elapsed-zero on every retarget. Our first implementation instead used a continuous, never-restarting exponential-decay loop (`current += (target-current)*(1-e^(-3dt/duration))`) - shaped similarly (fast-then-slow) but with a much weaker initial response than `expo.out` (initial rate `3/duration` vs `expo.out`'s `~6.93/duration`), so for the same declared duration it visibly lagged behind real GSAP, most noticeably right after a single scroll tick rather than at full settle. Fixed to match: `from`/`to`/`elapsed` state, `ease(elapsed/duration)` each tick, `elapsed` reset to 0 whenever `update()` retargets - drives `animation.totalProgress()` directly rather than via a helper `Tween`, since `Tween` is DOM-property-oriented and this is a single plain numeric value. Exposes **two** entry points, not one: `update()` (scroll-driven, smoothed once *settled*) and `snapTo()` (recalculation-driven, always instant). `update()` keeps snapping instantly - not smoothing - until this sync's own ticker tick has fired at least once since creation. This matters for a real bug: reloading mid-page, the browser's own async scroll-position restoration doesn't always land before this controller's first read of `window.scrollY`, so the very first read can be stale (e.g. 0), and the browser's restoration then fires its own native `scroll` event shortly after with the real position - if that were treated as an ordinary smoothed update, the tween would visibly "rewind" to the stale position and ease back into place on every reload, even though nothing the user did should have caused any animation. Read the long comment at the top of `createSmoothSync` before changing this - two mitigations are needed together (the `settled` gate here, plus `on-scroll.ts`'s `refresh()` always calling `snapTo()`) because neither alone is airtight given browser restoration timing isn't specified/guaranteed.
- **`on-scroll.ts`** ã `OnScroll` class + `parseEdge()`/`resolvePositionString()` (pure, exported, unit-tested independent of any DOM measurement) for `"<edge> <edge>"` position strings (`top|center|bottom|left|right|N%|Npx`, default `start: "top bottom"` / `end: "bottom top"`, plus a compound `"top+=100"`/`"30%-=20"` trailing-pixel-offset syntax on a single edge token). `resolvePositionValue()` special-cases a pure `"+=N"`/`"-=N"` string (no edge tokens) as **N pixels relative to `end`'s `relativeBase` parameter** (the already-resolved `startY`) rather than routing it through the edge parser ã this is the standard `end: "+=500"` idiom for expressing a sticky/sync distance directly in pixels regardless of the trigger's own height; **do not remove this branch** - without it, `"+=N"` falls through to `parseEdge()`, where `parseFloat("+=N")` is `NaN` (the `+`/`=` characters aren't valid for a plain number), silently resolving to `{ratio:0, offsetPx:0}` and collapsing `end` to essentially the same point as `start`. The visible symptom of that regression is dramatic, not subtle: the entire synced animation compresses into a ~1px scroll range, so it appears to fire instantly and the smoothed progress oscillates wildly between 0 and 1 from ordinary scroll jitter (looks exactly like a jittery/broken animation, not an off-by-a-little position bug) - this is exactly the bug a user hit with the classic `end: "+=${distance}"` horizontal-scroll-gallery pattern. `axis?: "x" | "y"` (default `"y"`) swaps every measurement (`triggerRect.top`/`.height` ã `.left`/`.width`, `scrollY` ã scroll-X, viewport height ã viewport width) via `axisRect()`/`resolvedAxis()` for a horizontally-scrolled track, without `parseEdge`/`resolvePositionString` themselves needing to know about axes at all (`EDGE_KEYWORDS` already maps both `top:0/bottom:1` and `left:0/right:1` to the same ratios). `refresh()` reverts any sticky element to its `"before"` baseline *before* measuring (so a currently-stuck element's transformed state can't corrupt its own measurement), resolves start/end and sticky distance, then calls `update(true)` (`instant`) so a recalculation always repositions synchronously and never animates as a side effect - see `sync.ts`. `update()` computes progress and fires `onEnter/onLeave/onEnterBack/onLeaveBack` based on scroll direction. When not syncing, the default (and *only* supported - no `toggleActions` string configurability) behavior matches GSAP's own default `toggleActions: "play none none none"`: **only** crossing into range going forward calls `.play()`; entering backward, leaving forward, and leaving backward do nothing to the animation by default (their callbacks still fire). This was a real bug too, in the other direction from most of the others: it initially called `.reverse()` on leaveBack, which is a plausible-sounding but wrong default - it makes a scroll-reveal animation silently undo itself every time the user scrolls back up past the trigger, which is not what "no toggleActions specified" means in GSAP. `onUpdate` only fires while `inside` the trigger's range, or on the exact frame of entering/leaving it - **not** unconditionally on every scroll event site-wide. Progress is clamped (constant) outside the range, so a scroll anywhere else on the page produces no actual change for this instance; firing onUpdate for those was a real reported bug (it fired continuously while scrolling through unrelated page content, long before/after the trigger's own range). `sticky` is validated as `true`/a selector/an `Element` right after resolution - anything else (e.g. a stray number) logs a warning and disables sticking for that instance instead of crashing inside `sticky.ts` with a confusing `getBoundingClientRect is not a function`. No batched cross-instance refresh ordering, no `anticipatePin`, no snap, no custom-scroller-proxy system beyond a plain `Element` or `window`, no `start: "+=N"` support (only `end`, since `start` has no prior resolved value to be relative to) ã documented Phase 1 scope cuts.

**A trigger constructed already behind the current scroll position (e.g. a page reload deep into a long scroll-reveal sequence) used to visibly replay its animation from scratch instead of showing it already complete.** `update()`'s default-toggle branch (`inside && !wasInside` ã `.play()`) doesn't distinguish "genuinely just crossed during live scrolling" from "was already past this point before we ever started watching" - a fresh `paused: true` tween is still frozen at totalTime 0 regardless of which case applies, so calling `.play()` on it always animates from the beginning. On a page reloaded mid-scroll, every trigger whose `start` already lies behind the restored scroll position would `.play()` at construction and visibly animate in, all at once, looking like the whole page's reveals are replaying on load. Worse: a trigger already past its *own* `end` too (common a few triggers back on a long page) never got `.play()`d at all, since `inside` requires `scrollY <= endY` - `onEnter` never fires, so it stays invisible forever. Fixed by reusing the `instant` flag `update()` already takes (rather than a separate one-shot "first ever" flag): whenever a recalculation (construction, `refresh()`, or - for a `syncTo` child - the sync source's own first post-construction update, which may itself be correcting a stale reading, see `boundOnSyncSourceUpdate` in `on-scroll.ts`) discovers `scrollY >= startY` already and the trigger wasn't already inside, `onEnter` fires but the animation jumps straight to `totalProgress(1)` instead of `.play()`ing - no visible replay, and it's correctly shown complete even when already past its own end too. Only applies to non-sync triggers (`!syncController`) - a syncing trigger already self-corrects via `refresh()`'s `syncController.snapTo()`. A genuine live-scroll crossing *after* construction is unaffected (`instant` is false for an ordinary scroll event), so the normal reveal animation still plays as expected. Covered by tests in `on-scroll.test.ts`.

**`syncTo` (GSAP's `containerAnimation` pattern - reveals nested inside a pinned/stuck, scrubbed track) is implemented.** `syncTo: someAnimation` ties this trigger's notion of "current scroll position" to `someAnimation.totalProgress()` instead of native scroll, and updates are driven by subscribing to the container's own `"update"` event (`Animation.on`) rather than `addScrollListener` - a synced container re-renders every ticker frame via its own smoothing, not just on native `scroll` events, so listening to *its* updates is what actually keeps a nested trigger in sync. `start`/`end` position strings for a `syncTo` trigger are resolved against the trigger element's **measured position at the container's totalProgress 0 and 1** (`measureSyncSourceEdges()`, via silent `seek()` calls so no stray `onUpdate`/events fire during the probe) rather than real scrollY/scrollX, producing a **progress ratio** (not a pixel value) stored in the same `startY`/`endY` fields - this assumes the container's own motion is linear (`ease: "none"`), which is the standard convention for a scrubbed container (verified against GSAP's real ScrollTrigger.js source: it also assumes/requires this, converting ratios through the container's own linear time-vs-scroll relationship). A relative `end: "+=N"` for a `syncTo` trigger is deliberately reinterpreted as **N pixels of the trigger element's own measured travel**, converted to an equivalent progress delta - simpler and more intuitive than GSAP's actual internal unit (seconds along the container tween's own duration axis, an implementation detail its real source exposes but that we don't try to replicate byte-for-byte). Debug markers are not supported in `syncTo` mode (not exercised by anything using this feature yet - a scope cut, not an oversight).

- **`markers.ts`** ã debug-only start/end lines (`debug: true`), isolated so it stays out of bundles that never enable it. Uses `position: absolute` at the *document*-Y coordinate (not viewport-relative), so it naturally scrolls with the page and only needs updating on refresh, not on every scroll event. Labels render **below** their line (not above via a negative `translateY`) - a marker positioned near the very top of the document would otherwise have its label pushed off-screen above `y: 0`. `vars.id` (`OnScrollVars.id`) shows up in the label text if provided, else a bare instance number, matching GSAP's own markers.

### SvgMotion (`src/plugins/svg-motion/`)

Three SVG effects bundled as one plugin, exposed through **one single public export, `SvgMotion(target, vars)`** (from `@six-js/core/SvgMotion`, see Public API below) ã `vars.mode: "draw" | "morph" | "path"` picks which effect runs. GSAP's DrawSVGPlugin + MorphSVGPlugin + MotionPathPlugin combined, per the actual ask rather than three separate plugins - but **not** exposed as three separately-named exports (`DrawSVG`/`MorphSVG`/`MotionPath`, this plugin's original shape): those names read as direct copies of GSAP's own paid Club GreenSock plugin product names rather than as six-js's own, a real brand-confusion/trademark concern once flagged, not a hypothetical one - the same category of problem `ScrollTrigger`ã`OnScroll`/`pin`ã`sticky`/`scrub`ã`sync` solved elsewhere in this codebase, just resolved here by collapsing to one name instead of finding three new ones (there was no non-derivative single word for each effect that wasn't either GSAP's own term or a worse, less-clear one). `svg-motion.ts`'s own top-of-file doc comment has the full rationale. The three per-effect implementations underneath are internal-only now (not re-exported from the public entry point) but otherwise unchanged: `drawSVG()`/`morphSVG()`/`motionPath()` (plain functions, still named this way internally - these names were never the actual public-facing risk, the capitalized product-name-shaped aliases were) each still **subclass `core/animation.ts`'s `Animation` directly** (exactly what `Tween` and `Timeline` themselves do), instead of going through the `PropertyTrack` system (`tween/property-track.ts`) or driving the ticker directly (`burst.ts`'s approach). None of the three fit a `PropertyTrack` (dasharray needs the path's total length up front; morph needs pre-sampled point sets; motionPath needs a live path query every render) - subclassing `Animation` gets play/pause/resume/reverse/restart/repeat/boomerang/delay/kill/context-capture/events/Timeline-nesting for free without adding a fourth track kind to core, and without modifying `property-track.ts`/`registry.ts`/`tween.ts` at all (every other plugin here follows the same "don't touch Tween/Timeline internals" rule).

- **`svg-motion.ts`** ã the dispatcher: `SvgMotion(target, vars)` switches on `vars.mode` and delegates to `drawSVG`/`morphSVG`/`motionPath`, stripping `mode` (and, for morph, pulling `toShape` out of `vars` into its own positional argument) before calling through. Return type is a discriminated overload keyed on `vars.mode` (`"morph"` ã `MorphAnimation`; `"draw"`/`"path"` ã their own `Animation | Timeline`), so callers get the correct narrowed type without a cast. An unrecognized `mode` throws rather than silently no-oping - a pure input-shape bug, not a runtime "target didn't resolve" condition (which still just warns + returns an empty `Timeline`, unchanged). `SvgMotionDrawVars`/`SvgMotionMorphVars`/`SvgMotionPathVars` (the three union members of `SvgMotionVars`) each add a `mode` literal on top of the corresponding internal `DrawVars`/`MorphVars`/`MotionPathVars` (renamed from `SvgDrawVars`/`SvgMorphVars`/the previous `SvgMotionPathVars` respectively, to free up the latter name for its new public meaning and drop the redundant `Svg` prefix now that they're internal-only, already inside `svg-motion/`).

- **`svg-geometry.ts`** ã `resolveGeometry()` resolves an `Element | string` shape input: an existing element is used as-is, a CSS selector that resolves to a geometry-capable tag uses that match, anything else (including a syntactically invalid selector, e.g. raw `"M0,0 L10,10"` path data) is materialized into a **detached** `<path>` (never appended to the DOM - per spec, `getTotalLength`/`getPointAtLength` work on detached geometry too) - this is what lets `motionPath`'s `path` and `morphSVG`'s `toShape` accept a raw `d` string with no visible/rendered guide element required, matching GSAP's own MotionPathPlugin. Tag-name-based (`isGeometryElement`), deliberately **not** `instanceof SVGGeometryElement` - that global doesn't exist at all in jsdom (the test environment), so an `instanceof` check would throw a `ReferenceError` the moment a test imports the module, not just fail the check. `shapeToPath()` converts `circle`/`ellipse`/`rect`(rounded or not)/`line`/`polyline`/`polygon` into an equivalent `d` string (a `path` is returned as its own `d` unchanged) so `drawSVG`/`morphSVG` can operate on any basic shape, not just literal `<path>` elements.
- **`draw.ts`** ã `DrawAnimation` animates `stroke-dasharray`/`stroke-dashoffset` to reveal (or hide) a shape's stroke - the classic "line draw" effect, e.g. drawing in a signature or a word made of letter outlines. `from`/`to` are each a ratio or an explicit `[start, end]` window (a single value means `[0, value]`, the ordinary fixed-start reveal; a two-token `"20% 80%"` moves both ends independently, for a moving "comet" segment). `drawSVG(target, vars)` resolves `target` like any other six-js target - resolving to more than one shape (e.g. `drawSVG("svg .letter", { stagger: 0.15 })` for a whole word) builds one `DrawAnimation` per shape and groups them in a `Timeline`, same "reuse `Timeline`, don't invent a group type" principle `six.to()`'s own stagger support uses.
- **`motion-path.ts`** ã `MotionPathAnimation` moves a target along a guide path (e.g. a plane following a wavy SVG curve) by writing into the **same shared per-element transform cache** every ordinary tween writes into (`getTransformCache`/`renderTransform` from `animate/transform-cache.ts`), not a competing `element.style.transform =` write of its own - so it composes cleanly with any other transform tween running on the same element instead of one clobbering the other. Applied as a **delta from the path's own start point** (matching GSAP's real MotionPathPlugin behavior): the target keeps whatever base x/y it already had, and moves by the same amount the path point itself moves relative to progress 0 - it does not jump to the path's absolute SVG coordinates, which are usually a totally different coordinate space than the target's own page position. `autoRotate: true | number` faces the target along the path's tangent (via a small symmetric before/after sample, not a forward-only difference, so the first/last frame still gets a meaningful direction); a number adds a fixed degree offset on top, to correct for whichever way the target's own artwork points by default.
- **`morph.ts`** ã `MorphAnimation` animates `target`'s own `d` attribute from its current shape into `toShape`'s (GSAP's MorphSVGPlugin) by sampling both shapes into equal-length point sets (`samplePoints()`, via `getPointAtLength`) and interpolating each pair positionally, reconstructing a smooth curve each frame via a uniform Catmull-Rom ã cubic-bezier conversion (`pointsToBezierPath()`) rather than a faceted polyline. **Not** GSAP's own shape-fitting/point-reduction algorithm - a documented scope cut; good enough for typical blob/icon/letterform morphs, but very different topologies may show simple straight-line-interpolation artifacts. `alignToFrom()` picks whichever rotation/winding-direction of the sampled `to` points best lines up index-for-index with `from` (minimizing total squared point distance) before interpolating - both shapes are sampled independently starting from their own path's start, which for two unrelated shapes usually land at very different physical spots/windings, so interpolating them unmodified visibly "twists" the morph (a real, easy-to-see artifact). Only tries rotations for **closed** shapes (open paths only try forward vs reversed - there's no "wrap around" on an open path). `samplePoints`/`alignToFrom`/`lerpPoints`/`pointsToBezierPath`/`isClosed` are all exported standalone (like `scroll-trigger.ts`'s `parseEdge`) so the point math is unit-testable against mocked geometry, independent of the class.

**Testing note**: jsdom implements **no SVG geometry methods at all** - `getTotalLength`/`getPointAtLength` don't exist on any SVG element regardless of tag or DOM attachment (unlike the layout-method gap documented below, this isn't even attachment-dependent). `vi.spyOn` requires the method to already exist, so tests assign it directly (`(el as unknown as {...}).getTotalLength = () => length`) rather than spying. For a raw `d`-string path input (materialized internally by `resolveGeometry`, so there's no element reference to pre-mock before construction), the only option is a shared `SVGElement.prototype` mock, cleaned up in a `finally` block - see `motion-path.test.ts`'s "accepts a raw path `d` string" test.

### SmoothScroll (`src/plugins/smooth-scroll/`)

A Lenis-modeled smooth-scrolling driver (`SmoothScroll`, from `@six-js/core/SmoothScroll` - not on the `six` object, see Public API below), adapted to six-js's own architecture/naming rather than ported 1:1. The one design decision everything else follows from: this moves the REAL native scroll position (`window.scrollTo`/`el.scrollTop`, via the exact same `on-scroll/observer.ts` helpers `OnScroll` itself reads through), not a `transform` on a wrapper/content pair - verified as what modern Lenis (v2) itself does by reading its actual source, not assumed from older writeups describing the transform-based approach. Consequences, all wanted: `OnScroll` needs zero changes/awareness of this class to keep working alongside it; native anchor links/find-in-page/`IntersectionObserver`/screen readers all keep working unmodified; and after every position write it also dispatches a synthetic native-shaped `scroll` `Event` synchronously (the real browser-originated one is asynchronous/queued), mirroring GSAP's own documented Lenis+ScrollTrigger integration guidance (`lenis.on('scroll', ScrollTrigger.update)`) so `OnScroll`'s progress doesn't lag a frame behind.

Deliberate departures from Lenis: `velocity` is real px/second (not Lenis's frame-rate-*dependent* raw per-update pixel delta); a content `ResizeObserver` (catches e.g. a lazy-loaded image growing the page) only refreshes the clamp limit rather than cancelling in-flight motion the way an explicit window resize does (real Lenis snaps unconditionally on every resize - fine for a rare window resize, but actively disruptive here since *some* element resizing mid-scroll from six-js's own Tween/Timeline is routine, not a rare edge case); touch scrolling is left fully native (matches Lenis's own `syncTouch: false` default, not a fabricated "safer" choice - a `touchmove` listener exists only to `preventDefault` while stopped/locked, e.g. so `.stop()` for a modal freezes the background on touch too); no `infinite` looping mode (needs duplicated DOM content, not needed for this plugin's own ask); method/event naming matches six-js's own vocabulary (`kill()` not `destroy()`, `scroller`/`axis` vars matching `OnScrollVars` exactly rather than Lenis's `wrapper`/`orientation`, `on()/off()` returning `this`, `ease` shaped like `TweenVars.ease` - a six-js ease name or function via `resolveEase`, not Lenis's function-only `easing`).

- **`motion.ts`** ã `ScrollMotion`, the per-frame numeric model: `lerp` (continuous damping toward a retargetable value, tuned to Lenis's own well-tested 60fps default feel) or `duration`+`ease` (a fixed eased tween instead) - `resolveMotionVars()` in `smooth-scroll.ts` switches to duration+ease mode if *either* is set on a call (matching Lenis's own ergonomics: `scrollTo(el, {duration: 1.5})` alone just works).
- **`wheel.ts`** ã `normalizeWheel()` cross-browser/cross-`deltaMode` wheel delta normalization. Horizontal instances accept whichever axis of input is larger (so an ordinary vertical mouse wheel still drives a horizontal gallery - a well-known affordance most mice/trackpads rely on) - verified against Lenis's own `gestureOrientation` handling rather than invented.
- **`scroll-to.ts`** ã `resolveScrollTarget()` for `SmoothScroll.scrollTo()`'s target shapes (element/selector/number/`"top"`/`"bottom"`, plus relative `+=N`/`-=N`).

`SmoothScroll` and `OnScroll` are two independently-usable plugins that happen to compose cleanly (see the design decision above) - `OnScroll`'s own `sync`/`syncTo` machinery has no dependency on `SmoothScroll` existing at all.

### Burst (`src/plugins/burst/`)

`Burst(origin, vars)` (from `@six-js/core/Burst` - not on the `six` object, see Public API below) - a firework/confetti-style one-shot particle effect, driving the shared `ticker` directly with real continuous 2D projectile motion (constant launch velocity + constant gravity, integrated every frame via `burst-physics.ts`'s `projectilePosition()`) rather than built on `Tween`/`Timeline` - matches GSAP's own `Physics2DPlugin` closed-form (no-friction) branch. An earlier version chained two eased `Tween` phases (an outward burst, then a fall) instead, which put a visible kink in the motion at the phase boundary (x-velocity eased to a dead stop instead of drifting continuously like a real launched object) - driving the ticker directly and writing the transform each frame from the exact physics formula removes that seam entirely.

Two particle sources, sharing one lifecycle rule (motion is never time-boxed - a particle keeps falling under gravity for as long as it genuinely takes to reach the viewport, and fading - if `fade: true`, the default - only ever starts once it's actually crossed below the viewport, never on a timer): **direct mode** (default, no `clone`) animates the resolved `targets` elements themselves in place - switched to `position: fixed` (pinned to their own exact pre-measured position first, so the switch is seamless) so they can genuinely fall off-page, never cloned/removed, left in the DOM once done; **clone mode** (`clone: N`) spawns `N` disposable clones of `targets` from `origin`'s position instead, removed from the DOM once finished. `container` (default `document.body`) decides clone positioning: `document.body` gets `position: fixed` (viewport-anchored) clones, anything else gets `position: absolute` clones confined to that container (nudged to `position: relative` first if it was `static`, so it actually establishes a containing block). All of a direct-mode burst's target rects are measured up front, before any of them are mutated - pinning the first target to `position: fixed` pulls it out of flow immediately, which can reflow later siblings (e.g. following characters sliding left to fill the gap), so a live re-measurement mid-loop would read an already-shifted position for anything measured after the first mutation.

### Parallax (`src/plugins/parallax/`)

`Parallax(target, vars)` (from `@six-js/core/Parallax` - not on the `six` object, see Public API below) - a mouse-move parallax effect: elements drift by different distances ("depth") in response to the cursor's position across the whole `window`, giving a layered depth illusion (e.g. a hero section where background shapes move less than foreground ones as the cursor moves). Each element's own depth is an `sx-parallax-strength="N"` attribute (px of offset at the extreme cursor position, i.e. a viewport edge; negative values invert the direction - e.g. a background layer drifting the OPPOSITE way from the cursor, for a deeper illusion; no clamping); `vars.strength` (default 30) is only the fallback for elements with no attribute of their own. Named `sx-*`, not `data-*`, to read consistently with the rest of the `sx-*` component/attribute family (`src/components/`'s `sx-marquee`/`sx-dialog`/`sx-slider` tags and their own plain, unprefixed-by-`data-` option attributes) rather than as a generic HTML custom-data attribute.

Architecturally a **hybrid of two existing plugin patterns with no prior exact precedent in this codebase**: driven directly off the shared `ticker` (`core/ticker.ts`) like `Burst` - an ongoing effect with no fixed duration, not an `Animation`/`Tween`/`Timeline` - but writing every frame into the SAME shared per-element transform cache (`animate/transform-cache.ts`) `motion-path.ts` uses, so a parallax offset composes additively with whatever other transform effect is already on the element (e.g. an `OnScroll` reveal also animating x/y), instead of clobbering it the way a raw `el.style.transform =` write (`Burst`'s own direct mode) would. **Composability caveat**: unlike a `Tween`, these writes are NOT tracked by `tween/overwrite.ts` (that system is `Tween`-only) - a `six.to(el, { x: ... })` running concurrently on the same element will silently fight with Parallax's per-frame writes for as long as both run; for `motion-path` this risk is minor since it's normally short-lived, but for Parallax it's more consequential since it has no fixed duration and can run indefinitely.

Smoothing uses the exact same `damp(from, to, lambda, deltaSeconds)` frame-rate-independent decay formula as `smooth-scroll/motion.ts`'s own `lerp` option (`lambda = lerpAmount * 60`, `vars.lerp` default `0.1`, `0`/omitted means an instant 1:1 follow) - duplicated locally in `parallax-motion.ts` rather than imported: every one of six-js's published plugin subpaths is built as a fully independent bundle (see "Public API"/UMD below) specifically so no subpath pays for another's code, and there are zero cross-plugin-folder source imports anywhere in this codebase today (confirmed by grep) - not worth being the first exception for a 3-line pure function. Unlike `Burst`'s closed-form absolute-time physics (constant velocity/gravity, no per-frame delta needed), Parallax's damping *target* (the cursor) moves unpredictably, so it genuinely needs `deltaMs` accumulated every tick, same as how `smooth-scroll.ts` itself drives its own `ScrollMotion.advance(deltaMs / 1000)`.

The shared cursor position is read from `parallax-pointer.ts`'s refcounted singleton - one native `mousemove` listener total, no matter how many `Parallax()` calls are active - modeled on `on-scroll/observer.ts`'s `addResizeListener`/`removeResizeListener` (lazy-attach on first use, detach on last release), **not** `sticky.ts`'s per-element `WeakMap` cache, since a mouse position is one single global resource, not something keyed per DOM element. Deliberately poll-based (`getPointerPosition()`), not push-based (no stored subscriber callbacks) - a real, intentional divergence from `addResizeListener`'s shape despite being modeled on it: every `Parallax()` instance already ticks every frame via its own `ticker` listener and can just read the current position there, so there's nothing worth notifying on every individual mousemove.

`kill()` resets every managed element back to its snapshotted base x/y, rather than leaving it at its last lerped offset - a deliberate divergence from `Burst`'s "leave direct-mode elements wherever they landed" default. Burst's trajectory *is* the effect, so freezing it in place on kill is correct; Parallax's offset is a decorative wobble meant to be toggled on/off (mounted/unmounted repeatedly in a SPA - a route change, a modal closing), and resetting to base keeps repeated mount/unmount cycles idempotent instead of baking in whatever drift existed at the moment of the last mouse move before unmount.

Never measures layout (no `getBoundingClientRect()` anywhere) - it only applies a floating transform delta on top of whatever transform state already exists, so unlike `on-scroll`/`sticky` it's fully testable in jsdom with no rect-mocking gap.

**Performance**: the `ticker` listener itself is always-on for as long as an instance is alive (same "kept always-on, no autoSleep" trade-off `core/ticker.ts`'s own `rootTimeline` accepts), but the per-element work inside it isn't - each managed element is skipped entirely once it's already sitting on its current target, checked BEFORE touching the cache, not after. A still cursor means every element converges once and then does zero further work every subsequent frame (no `damp()` call, no `style.transform =` write, no style recalc/composite) for as long as the mouse stays still; only a cursor move (which changes the target) or an element still mid-chase re-enters the write path. Without this guard, an idle mouse would still force a real DOM write on every managed element on every single frame indefinitely. Covered by a regression test in `parallax.test.ts` (`vi.spyOn(el.style, "transform", "set")` asserting zero calls across several settled ticks, then confirming writes resume the instant the cursor moves again).

Scope cuts (v1): whole-`window` `mousemove` only, no per-container/hover-scoped mode; no touch/device-orientation input; no `pause()`/`resume()` (`kill()` only, matching `BurstController` exactly - there's no `Animation` base class here to inherit them from, and half-implementing them isn't worth it); not captured by `six.context()` scopes (matches `Burst` - neither is an `Animation`/`Timeline`); not on the `six` object, subpath-only import like every plugin except `OnScroll`'s convenience field.

### ScrambleText (`src/plugins/scramble-text/`)

`ScrambleText(target, vars)` (from `@six-js/core/ScrambleText`, aliasing the plugin's own `scrambleText` function - not on the `six` object, see Public API below) - GSAP's `ScrambleTextPlugin`, reimplemented from its public documented behavior (it's a paid Club GreenSock bonus plugin - no source is publicly readable, unlike the free plugins this codebase has verified byte-for-byte elsewhere). A plain `Animation` subclass (`ScrambleTextAnimation`) rather than a `Tween` property, for the same reason `SvgMotion`'s three effects are: the visible per-frame state (which characters are already-correct vs still-scrambled, how many total characters are even displayed) isn't a single interpolated number/color/string the `PropertyTrack` system already models. Reveal math: `displayLen - revealCount` (remaining scrambled filler characters) reduces algebraically to `oldLen * (1 - eased)` regardless of `newLen` - what makes `tweenLength` (the old text's visible length shrinking to 0 while the new text's revealed portion grows) fall out of the same two GSAP-documented formulas (`revealCount` grows toward `newLen`, `displayLen` interpolates oldãnew length) rather than needing a third, separate "shrink the old tail" rule. `mode: "odometer"` (`OdometerAnimation`, `odometer.ts`) is a **six-js addition, not part of GSAP** - each character spins on its own vertical reel and stops on the correct letter, with `charStagger` producing a left-to-right (or reversed) landing wave instead of every reel stopping in lockstep.

- **`char-sets.ts`** ã `resolveCharSet()`, named character pools (`upperCase` default, etc.) for the scrambled/filler portion.
- **`text-units.ts`** ã `splitUnits()`/`randomChar()`/`randomString()`. `delimiter` decides the reveal granularity: `""` (default) splits/reveals per-character, `" "` per-word (GSAP-documented).
- **`reveal.ts`** ã `delayedProgress()`, the shared `revealDelay` (pure scrambling before any reveal begins) timing curve both modes use.
- Resolving to more than one element builds one animation per element and groups them in a `Timeline` (same "reuse `Timeline`, don't invent a group type" principle every other multi-target six-js effect uses).

### SplitText (`src/plugins/split-text/`)

`SplitText` (from `@six-js/core/SplitText` - not on the `six` object, see Public API below) - GSAP's `SplitText`, splits into `chars`/`words`/`lines` (any combination via `type`). Unlike every other plugin in this codebase, it has **no** `Animation`/`Tween`/`Timeline` involvement at all - it's a pure DOM-restructuring utility (wrap text in spans); animating the resulting `.chars`/`.words`/`.lines` arrays is left entirely to an ordinary `six.to()`/stagger. Idempotent per-element ownership: splitting an element that's already owned by a previous (unreverted) `SplitText` instance auto-`kill()`s that instance first (`ownerRegistry`, a `WeakMap<Element, SplitText>`), rather than doubly-wrapping or leaking the old spans. `revert()` restores each element's original `innerHTML`/`aria-*` attributes verbatim (recorded per-element on `split()`) rather than trying to algebraically un-wrap the generated spans. `overflow` (mask a `chars`/`words`/`lines` collection in a clipped wrapper, for a reveal-from-behind-a-mask effect) reuses whichever collection `type` already produced rather than re-splitting. `type: "lines"` additionally watches for resize (`auto-resplit.ts`) and re-splits automatically, since line breaks are layout-dependent and can change on viewport/font/container width changes - `chars`/`words` splits never need this, their boundaries don't depend on layout.

### Components (`src/components/`)

Three independent `sx-*` custom elements (`sx-marquee`, `sx-dialog` + its trigger/pull/close-cursor helpers, `sx-slider` + its track/slide/pagination/nav sub-elements) - **deliberately architecturally independent of the core *animation* engine**: none of them use `Animation`/`Tween`/`Timeline`/`rootTimeline`, `six.config()`/`setDefaults()`, `overwrite.ts`, or `six.context()`. The one shared low-level exception is `core/ticker.ts`'s own `ticker` singleton - `sx-marquee` and `sx-slider` (`slider-track.ts`) both register a listener on it directly (`ticker.add(...)`) for their per-frame motion, rather than running an independent `requestAnimationFrame` loop of their own; they just never go through `Animation`/`Tween`/`Timeline` to get there. `sx-slider`'s drag/momentum additionally uses its own `InertiaPhysics` (`core/inertia-physics.ts`); `sx-dialog` uses CSS transitions/effects plus its own scroll-lock, also with no ticker involvement. The boundary that *is* deliberate: components stay usable/predictable standalone, without pulling in or being affected by the animation engine's own global state (defaults, active `overwrite` tracking, an active `context()` scope) - sharing the ticker's timing source doesn't compromise that, since the ticker itself carries no such state. Each element registers itself idempotently (`customElements.get(tag) ?? customElements.define(tag, ...)`), so calling a `register*()` more than once - or both an individual one and the umbrella `registerComponents()` - is always safe.

**`sx-animate` was removed entirely** (used to live at `src/components/animate/`) - the same fade/slide-in-on-scroll effect is just an ordinary `six.to()` tween driven by `OnScroll`/`IntersectionObserver` from userland; keeping a whole separate custom element for it duplicated what the core engine already does well. If anything elsewhere in this file still mentions `sx-animate`/`SxAnimate`/`registerAnimate`, it's stale - flag it.

**None of the three use Shadow DOM** - `sx-marquee` briefly did (its original, pre-six-js-rewrite implementation), and `sx-dialog`/`sx-slider` were both converted TO shadow DOM and then reverted BACK within this same rework, deliberately: **React compatibility was the deciding factor**. React's event system has real, well-documented friction with shadow DOM (event retargeting across the shadow boundary changes what `event.target` looks like to a listener outside it, which can break React's own event delegation depending on version/configuration) - since these components need to work when rendered from/around React trees, not just plain HTML, staying light-DOM-only avoids that class of bug entirely rather than working around it case by case. All three render their own internal chrome (`sx-dialog`'s backdrop/core wrapper, `sx-slider`'s none at all - see below, `sx-marquee`'s `.container` wrapper) as **real light-DOM children**, built via plain `innerHTML`/`appendChild`, not a `<slot>`-based composition. `sx-marquee`'s `render()` is idempotent about this (checks for an existing `:scope > .container` before wrapping again), matching `dialog.ts`'s own `originalContentHTML`-capture-and-reinsert pattern for the same "don't double-wrap on a second render" reason. `sx-slider` needed **no internal wrapper at all** - `sx-slider-track`/`sx-slider-slide` are the user's own authored markup (not chrome `sx-slider` generates), so every one of `slider.ts`'s ~30 internal methods (`this.querySelector("sx-slider-track")`, drag physics, breakpoints, etc.) works completely unmodified, light-DOM-first exactly as originally written.

**CSS is a real, separate, manually-imported file - never auto-injected by JS.** Every component's `.css` file (`dialog.css`, `slider.css`, `marquee.css`, all light-DOM selectors like `sx-dialog > .sx-dialog-core`, no `:host`/`::slotted`) is an ordinary stylesheet, combined by Vite into one published `dist/components.css` - a fixed name, not a content hash, so the path is predictable/documentable. Getting that exact filename out of Vite took two attempts, both documented in `vite.config.ts`: library-mode CSS output ignores `rollupOptions.output.assetFileNames` outright (still emits `style.css`, confirmed against a real build), and even renaming the asset from a `generateBundle` hook didn't stick (something in Vite's own CSS pipeline runs after that hook and re-asserts the name, also confirmed against a real build) - what actually works is a plain filesystem rename from `closeBundle`, which fires only once Vite has fully finished writing every file to disk, too late for anything else in Vite to undo it. **Using any `sx-*` tag requires the consumer to explicitly `import "@six-js/core/components.css"`** (or a plain `<link rel="stylesheet">`, e.g. the demo pages' own `<link rel="stylesheet" href="./dist/components.css">`) - `registerX()` does **not** inject or link any CSS itself, on purpose: this is the plainest, most standard "ship a CSS file, the consumer imports it once" convention (the same one virtually every non-CSS-in-JS component library uses), works identically in a plain HTML page, a bundler-based app, or a React app, and needed no runtime JS-DOM-injection machinery at all (an earlier version of this used Vite's `?inline` query to embed the CSS as a JS string and auto-`<link>`/`<style>` it in on first `registerX()` call - deliberately reverted in favor of this simpler, more conventional approach). `dialog/index.ts`/`slider/index.ts`/`marquee/index.ts` each still have a plain `import "./dialog.css"` (etc.) - that's solely so Vite's own build discovers and bundles the CSS into `dist/components.css` in the first place; it has no runtime effect of its own (a bare CSS import produces no JS side effect, it's purely a build-time signal).

Shares a few small, animation-engine-independent utilities under `src/core/`/`src/properties/` with each other (not with `Tween`/`Timeline`): `safe-element.ts` (an SSR/no-DOM-safe `HTMLElement` base class), `time.ts` (`parseTimeValue()`, attribute-string ã milliseconds), `observer.ts` (a shared `IntersectionObserver` wrapper - `enter`/`leave` callbacks, unrelated to `on-scroll/observer.ts`'s scroll observer despite the matching filename/one-directory-up path), `properties/transform-state.ts` (`setTransformValue()`/`buildTransformString()`, `sx-marquee`/`sx-slider`'s own transform-string builder - unrelated to `animate/transform-cache.ts`), and `core/container-breakpoints.ts` (`ContainerBreakpoints`, `sx-slider`/`sx-dialog`'s own per-breakpoint options parser, keyed on container width - unrelated to `core/breakpoint.ts`'s media-query-based `Breakpoint`, see its own entry above).

There is no longer a package-level `enableElements()`/`six.enableElements()` convenience - `src/api/six.ts` deliberately does not import anything from `src/components/` at all (previously it did, purely to support that one convenience function, which meant the entire components tree was forced into the core `index.js` chunk even for consumers who never called it); registering components now always goes through the `@six-js/core/Components` subpath explicitly (either `registerComponents()` for all three, or the individual `registerX()` you need). Since registration is fully opt-in per component, forgetting one is easy and fails **silently** by default - an unregistered `<sx-*>` tag is just an inert, unstyled `HTMLUnknownElement`, no browser error or warning. Mitigated by `core/unregistered-elements.ts`'s `watchForUnregisteredElements()` (see Core above) - **not** a components-side file, deliberately: a first attempt lived in `src/components/shared/` and hardcoded the known tag names, triggered from each `registerX()` - but that can only ever fire if at least one `registerX()` runs, so it silently missed the single most common real case, a page that imports `six` and drops `<sx-marquee>` in its markup but never touches `@six-js/core/Components` at all (confirmed live via `test.html`). Moved to core and made tag-pattern-generic (any `sx-*` tag, no hardcoded list) specifically so it runs unconditionally for every consumer regardless of which subpaths - if any - they've imported.

- **`marquee/marquee.ts`** ã `SxMarquee` (+ `SxMarqueeInner`/`SxMarqueeItem`, `marquee/index.ts`). Continuously offsets a transform each ticker frame, cloning content once (`clone` attribute, default on) to cover the loop seam.
- **`dialog/`** ã `SxDialog` + `SxDialogTrigger`/`SxDialogPull`/`SxCloseCursor`. Background scroll lock is two independent, composable levels per open dialog's own `scrollable` option (`false`: CSS `overflow: hidden` on `document.documentElement` - deliberately the root, not `<body>`, since `<body>`'s auto height means `overflow: hidden` on it alone usually clips nothing - **plus** wheel/touch `preventDefault`, both needed together since `overflow: hidden` alone can't stop a JS-driven scroller like `SmoothScroll` moving the page via explicit `scrollTo()` calls; `"scrollbar"`: `preventDefault` only, scrollbar stays visible; `true`: no lock at all) - tracked via a static `openStack` so multiple simultaneously-open dialogs share one correctly-composed lock instead of each fighting over it independently. Drag-to-dismiss direction/sign is looked up per `position` (`DRAG_MAP`) rather than hardcoded, so e.g. a `position: "top"` dialog drags up-to-dismiss while `position: "bottom"` drags down-to-dismiss.
- **`slider/slider.ts`** ã `SxSlider` (+ track/slide/pagination/prev/next/progress sub-elements, `slider-registry.ts`). By far the largest component (~1,350 lines) - loop and non-loop modes track position differently: non-loop tracks a pagination-dot **step count**, deliberately not derived from the raw slide index, since a boundary-clamped move can land between two dots with no exact match, and a "nearest stop" lookup at that position would pick the dot being moved *away* from rather than toward; any direct jump (`goTo`/sync/drag-snap/initial index) instead recomputes the step fresh via `findStopIndex()`, since there's no prior "step" to preserve there.

### Public API (`src/api/six.ts`, `src/index.ts`, `src/entries/`)

The package is split for tree-shaking: `import { six } from "@six-js/core"` (the `"."` export, built from `src/index.ts`) gives only `six.to/from/fromTo/set/timeline/config/context/breakpoint/utils` ã core, no plugin code. `six.to()`/`six.timeline()` attach the created `Tween`/`Timeline` to `rootTimeline`; with `stagger`, wraps the per-target tweens in a `Timeline` (returned to the caller) rather than a bare array, so the whole staggered group can still be paused/reversed/killed as one unit ã same "reuse `Timeline`, don't invent a group type" principle Timeline's own stagger support uses.

Every plugin is a separate subpath, each its own Vite lib entry (`vite.config.ts`) and its own `package.json` `exports` key, so a consumer who never imports a subpath never pays for its code:

- `@six-js/core/OnScroll` ã `{ OnScroll }` (`src/entries/on-scroll.ts`). **Exception to the "core has zero plugin code" rule**: `SixTweenVars`/`SixTimelineVars`'s `onScroll` convenience field (`six.to(el, { onScroll: {...} })` ã internally calls `OnScroll.create(...)`) needs `OnScroll` at the point `six.to()` is defined, so `on-scroll.ts` is still statically imported by `src/api/six.ts` and ships inside the core `index.js` chunk regardless of whether this subpath is also used. This subpath is an explicit/alternate import path (for calling `OnScroll.create()` directly), not a size reduction ã a deliberate, accepted trade-off, not an oversight.
- `@six-js/core/SmoothScroll` ã `{ SmoothScroll }` (`src/entries/smooth-scroll.ts`)
- `@six-js/core/SplitText` ã `{ SplitText }` (`src/entries/split-text.ts`)
- `@six-js/core/ScrambleText` ã `{ ScrambleText }` (`src/entries/scramble-text.ts`) ã re-exports the plugin's own `scrambleText` function under the capitalized binding name to match this subpath's name; the internal function name is untouched.
- `@six-js/core/Burst` ã `{ Burst }` (`src/entries/burst.ts`) ã same capitalized-alias pattern as `ScrambleText`, aliasing `burst`.
- `@six-js/core/SvgMotion` ã `{ SvgMotion }` (`src/entries/svg-motion.ts`) ã a single dispatch function (`SvgMotion(target, { mode: "draw" | "morph" | "path", ... })`), not three separate exports; see the SvgMotion section above for why (the original three-export shape's names read as copies of GSAP's own paid plugin product names).
- `@six-js/core/Parallax` ã `{ Parallax }` (`src/entries/parallax.ts`) ã same capitalized-alias pattern as `Burst`/`ScrambleText`, aliasing `parallax`.
- `@six-js/core/Components` ã `{ registerMarquee, registerSlider, registerDialog, registerComponents }` (`src/components/index.ts`, which is also this entry directly). Each `registerX` is an independent named export with no top-level call site tying them together, so importing only the ones you need lets a bundler's ordinary unused-export elimination drop the rest ã no per-component chunking or dynamic-import autoloader needed. `registerComponents()` (registers all three) is a no-tree-shaking convenience for anyone who doesn't care about size - there is no separate `six.enableElements()`/package-level convenience anymore (see the Components section above for why that was removed). `@six-js/core/components.css` is the one combined, real stylesheet all three components' own CSS is built into - **required**, not automatic: a consumer must `import "@six-js/core/components.css"` (or `<link>` it) themselves for any `sx-*` tag to be styled at all, since `registerX()` deliberately does not inject or link any CSS on its own (see Components above for why).

Every export binding across every subpath is PascalCase (`OnScroll`, `SmoothScroll`, `SplitText`, `ScrambleText`, `Burst`, `SvgMotion`) except `Components`' `registerX`/`registerComponents` functions, which stay camelCase-verb-shaped on purpose - they're actions, not classes/types, matching ordinary JS convention (`document.addEventListener`, not `Document.AddEventListener`).

`package.json`'s `sideEffects` is an explicit array (`dist/index.js`, `dist/six-js.umd.js`, `dist/Components.js`) rather than a blanket `true` ã every other plugin chunk has no top-level side effects (only function/class definitions), so bundlers can drop an entirely-unreferenced one outright.

**Every subpath except `.` had a `"types"` condition pointing at a `.d.ts` file that never actually existed on disk** ã a `tsc`/editor-only failure (the built `.js` was always correct; a consumer's bundler/browser never saw the problem), found from a downstream consumer (`six-js-docs`) hitting exactly six "cannot find module" errors, one per non-core subpath. Root cause: `vite-plugin-dts` mirrors each entry's path relative to the shared `src/` root into `dist/` ã `src/entries/on-scroll.ts` ã `dist/entries/on-scroll.d.ts`, `src/components/index.ts` ã `dist/components/index.d.ts` ã but `package.json`'s `exports` map promises a flat `dist/OnScroll.d.ts`/`dist/Components.d.ts` per subpath, matching the JS/UMD output names. The two were never reconciled; `.` alone happened to work because `src/index.ts` is already zero levels deep. First fix attempt was `rollupTypes: true` (the documented way to collapse an entry to one self-contained `.d.ts`) plus the `@microsoft/api-extractor` peer dep it requires ã reverted, since it still didn't emit files at the promised flat paths with a named multi-entry `build.lib.entry` object (suspected `vite-plugin-dts@5.0.x` bug with that combination). Fixed instead with `flattenTypesOutput()` (`vite.config.ts`) ã a `closeBundle` plugin, same shape/timing as `renameCssOutput` right below it ã that moves each mirrored shim to its flat name and rewrites its relative `from "..."` specifiers to still resolve from the new location; safe because every one of these generated files is a thin 1ã3 line re-export shim, not a real bundled declaration file. `libEntries` is now defined once and shared between `build.lib.entry` and `flattenTypesOutput()` so the two can't drift apart again.

**CDN/`<script>` usage (no bundler) gets one self-contained UMD file per subpath, each its own named global** - `scripts/build-umd.mjs` (run by `npm run build` after the main ES `vite build`), not a single `vite.umd.config.ts` anymore. UMD fundamentally can't be code-split (no cross-file `<script>` imports in the classic global-variable model), so each of the 9 targets (`six`, `OnScroll`, `SmoothScroll`, `SplitText`, `ScrambleText`, `Burst`, `SvgMotion`, `Parallax`, `Components`) is its own fully independent Vite build with its own copy of whatever core code it needs - looped over Vite's JS build API rather than 9 near-identical config files. Each has a dedicated entry under `src/entries/umd/` (distinct from the ESM entries under `src/entries/`) specifically so its export shape can be UMD-appropriate: a lone `export { x as default }` so Rollup assigns the global directly (`<script src=".../OnScroll.umd.js"></script>` then just `new OnScroll(...)`/`OnScroll.create(...)`, no wrapper object to destructure) for anything with one obvious primary export (`six`, `OnScroll`, `SmoothScroll`, `SplitText`, `ScrambleText`, `Burst`, `SvgMotion`, `Parallax` - `SvgMotion` moved into this bucket once it collapsed to a single dispatch function, see its own section above), or a plain named-exports object for `Components`, the one target with several unrelated exports (`{ registerMarquee, registerSlider, registerDialog, registerComponents }`, used as `Components.registerDialog()`). `package.json`'s `exports` map gives every subpath a `require` condition pointing at its own `.umd.js` alongside the existing `import` (ESM); Components' UMD build's own incidental CSS output is deleted rather than published under a second name - it's built from the exact same source files as the canonical `dist/components.css` the main build already produces, so there's nothing a duplicate would add.

**Real, accepted trade-off, not a bug if noticed while debugging**: loading more than one of these UMD files on the same page (e.g. `six-js.umd.js` *and* `OnScroll.umd.js`) means each bundled copy of `core/ticker.ts`'s singleton is a *separate* `Ticker` instance with its own independent rAF loop, not the literal same object every ESM import of `@six-js/core` would share. Both still tick at ~60fps, so nothing breaks outright, but they're not frame-synchronized with each other the way two ESM imports always are - documented in the build script itself, not just here.

## Testing

`vitest` + jsdom. `Ticker` tests always construct `new Ticker({ manual: true })` (never the shared `ticker` singleton) to avoid real rAF timing. Tests against `rootTimeline`/`ticker` (in `api/six.test.ts`) capture a starting point (`rootTimeline.totalTime()`) and only assert relative to it, since `rootTimeline` is a real module-level singleton shared across every test in the process ã asserting against absolute values makes tests order-dependent.

**Mock `window.scrollY`/`window.innerHeight` via `vi.spyOn(window, "scrollY", "get")`, never `Object.defineProperty`** ã `vi.restoreAllMocks()` in `afterEach` does not undo a raw `Object.defineProperty` override, so it silently leaks into later tests in the same file (looks exactly like a real caching bug when it's actually test-isolation; this cost real debugging time once already ã see `on-scroll.test.ts`/`six.test.ts`). Similarly, when manually simulating a scroll event that should invalidate `observer.ts`'s read cache, call `invalidateReads()` directly rather than relying on the dispatched event to trigger it ã there's no listener registered to catch the event until an `OnScroll` has actually been constructed, so `scrollTo()`-before-construction (setting up the initial position) would otherwise silently no-op.

jsdom does not implement layout: `getBoundingClientRect()` returns all-zero rects and `clientWidth`/`clientHeight`/`scrollHeight` are always 0 unless a test explicitly mocks them (`vi.spyOn(el, "getBoundingClientRect").mockReturnValue(...)`, as `on-scroll.test.ts` and `sticky.test.ts` do). The `%` unit-conversion path in `animate/unit-convert.ts` is untestable in jsdom for this reason and is only exercised by the browser smoke test (`index.html`).

`src/components/` (the independent `sx-*` web components, see their own Architecture section) has **no automated tests at all** - confirmed by directly checking for `.test.ts` files there (none exist), not assumed. Everything else in this codebase has unit/integration test coverage at minimum (several plugins explicitly still lack real-*browser* verification, see their own sections, but all have jsdom tests); the web components have neither. A real, current gap, not a stylistic choice - flagged here rather than silently glossed over.

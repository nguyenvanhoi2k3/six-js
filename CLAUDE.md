## Commands

```bash
npm run build       # gen-version + vite build (ES, single entry) + vite build --config vite.umd.config.ts -> dist/ (es + umd + type declarations)
npm run dev         # gen-version + vite build --watch (ES only) - rebuilds dist/ on every source change, doesn't serve anything itself
npm run sandbox     # vite dev server for index.html, importing /src/index.ts directly (TS transpiled on the fly) - an alternative to the Live Server workflow below
npm test            # vitest run (jsdom environment)
npm run test:watch  # vitest, watch mode
npm run typecheck   # tsc --noEmit
```

`npm run gen-version` regenerates `src/version.ts` from `package.json`'s `version` field — do not hand-edit `src/version.ts`.

`index.html` (project root) is a live sandbox exercising every Phase 1 piece against the real `six` API. It imports `./dist/six-js.es.js` (the **built** output), specifically so it can be opened with a plain static file server — VS Code's "Live Server" extension is the intended workflow here, not `npm run sandbox`. **Run `npm run build` (one-shot) or `npm run dev` (rebuilds on every save) before/while using Live Server**, or the sandbox is testing stale code. Do not change `index.html` back to importing `/src/index.ts` directly — a generic static server can't transpile TypeScript, and (worse) maps the `.ts` extension to the MPEG transport-stream video MIME type rather than JavaScript, so the browser rejects it with `Failed to load module script: ... MIME type of "video/mp2t"` under strict module-script MIME checking. Verified working via a plain static server (`serve`) with zero console/request errors after switching to the `dist/` import.

## Status: Phase 1 rewrite in progress

This codebase was rebuilt from scratch (see git history starting at "Rebuild Phase 1 core foundation") after an earlier attempt was abandoned for architectural problems — specifically: lifecycle state lived in a wrapper external to the thing being played (so nested timelines lost repeat/pause/reverse), time was accumulated incrementally per-instance instead of derived by formula (fragile under seeking/timeScale), and the same feature (stagger) got reimplemented three different ways. The rewrite's core design decision is one shared `Animation` base class (`core/animation.ts`) for both `Tween` and `Timeline`, with time flowing as a pure coordinate-transform down the tree rather than being accumulated anywhere except the `Ticker`'s own clock. The full reasoning is in the commit messages (each Phase 1 module landed as its own commit, several documenting a real bug the accompanying tests caught); the short version is inlined as comments at the relevant call sites below.

**A real lesson from this rewrite, worth not repeating**: several early ScrollTrigger bugs (pin snapping to the viewport top instead of its natural position, the default toggle behavior calling `.reverse()` on leaveBack) came from inventing a plausible-sounding default instead of checking GSAP's actual documented/known behavior first. For anything GSAP-equivalent - a default value, an edge-case behavior, what a specific callback should and shouldn't fire for - verify against GSAP's real behavior before implementing, rather than designing what "seems reasonable." An explicit audit pass cross-checked `scroll-trigger.ts`'s remaining defaults against known GSAP behavior (default `start`/`end`, `scrub` semantics, `markers` default, string `scroller` support, ease still applying under scrub, direction-based onEnter/onEnterBack, inclusive range boundaries) and found them already correct; the only other gap found was cosmetic (markers had no visible "start"/"end" text, since fixed).

**A second audit pass covered Core/Tween/Timeline/Animate** (not just ScrollTrigger) and found six real, verified-in-a-real-browser-or-by-test bugs, none of which needed any GSAP-behavior guessing - they were internal consistency bugs, provable from the code's own stated invariants:

1. **`pause()` then `play()` later jumped the animation ahead by however long it was paused, instead of resuming from where it froze.** Root cause: a paused child is simply skipped during its parent's render pass (so its `_tTime` stays frozen - correct so far), but its `_start` (position within the parent's coordinate space) was never adjusted. Once resumed, the very next render recomputed `(parentLocalTime - _start) * timeScale` using the parent's now-much-later local time against the same old fixed `_start` - i.e. it computed "where would this be if it had been playing the whole time," not "where it actually was." This is not a cosmetic issue: it breaks pause/resume for **any** animation attached to the always-ticking root timeline, which includes every top-level `six.to()`/`six.timeline()` AND, critically, every non-scrub ScrollTrigger animation (`vars.animation.pause()` runs immediately at creation; `.play()` fires whenever the user later scrolls into range) - unfixed, a `toggleActions: "play..."` reveal would have snapped straight to its end state the instant it was triggered, with no visible animation at all, on any page that had been open for longer than the tween's own duration before the trigger fired. Fixed via a new `AnimationParent._childResumed(child)` hook (`core/animation.ts`) that `Timeline` implements (`timeline.ts`): on the paused->playing transition, it re-anchors `child.startTime()` so the timeline's current local time maps back to exactly the child's frozen totalTime. Skipped only when the timeline's playhead hasn't reached the child's *originally scheduled* start yet (a child still waiting its turn in a sequenced timeline) - deliberately **not** gated on "has the child's totalTime already progressed past 0," which gets the single most common real case (a `paused: true` tween created and resumed later, per above) backwards, since such a tween's frozen totalTime is exactly 0 right up until the moment it's finally played. Verified with a jsdom unit test reproducing the exact scenario (`timeline.test.ts`) AND an end-to-end real-browser Playwright check (a tween created `paused: true`, left alone on the real ticker for 1.5 real seconds, then `.play()`d - confirmed it plays smoothly from 0% rather than jumping to 100%).
2. **`restart(includeDelay)` always replayed the delay, regardless of the flag** - `includeDelay: false` (the default, matching GSAP) is supposed to *skip* the delay on replay; only `includeDelay: true` should wait through it again. The code passed `totalTime(includeDelay ? -delay : 0, true)` - but `render()` unconditionally clamps negative totalTime to 0, so `-delay` and `0` land on the exact same value, making the two modes indistinguishable, and both had the *wrong-for-the-default-case* behavior (0 is the very start of this engine's delay window, since totalTime `[0, delay)` represents "waiting" on this animation's own axis - see `render()`'s doc comment). Fixed to `totalTime(includeDelay ? 0 : this._delay, true)` - skip-delay jumps straight to `_delay` (the boundary where the active portion begins), replay-delay goes to the true `0`. Covered by two new tests in `animation.test.ts`.
3. **A tween ending with a nonzero `z` (or having ever touched `rotationX`/`rotationY`) visibly snapped away its z-offset the instant it settled at rest.** `transform-cache.ts`'s `composeTransform()` chose between `translate3d(x,y,z)` and plain `translate(x,y)` based solely on `use3D` (true only while a tween is actively mid-flight, for GPU-compositing reasons) - at rest, `use3D` is false, so it always rendered `translate(x,y)`, which has no `z` component at all, silently discarding it. Fixed so a nonzero `z` always forces `translate3d`, regardless of `use3D` - `use3D` now only controls the compositing-layer promotion for the common `z === 0` case, never a license to lose real state. Covered by two new tests in `transform-cache.test.ts`.
4. **`restart()` on an animation that had already finished playing naturally (never explicitly paused) left it snapping straight back to its completed/end state instead of actually replaying.** `_childResumed` re-anchoring (see bug 1) only fires from `paused()`'s paused->playing transition (`resuming = value === false && this._ts === 0`); a tween/timeline that reached the end of its own duration without ever being paused still has `_ts !== 0` the whole time, so `restart()`'s `totalTime()` reset was silently overwritten on the very next render by the parent (e.g. the always-ticking root) recomputing `(parentLocalTime - _start) * timeScale` against the same stale, long-past `_start` - clamping straight back to "complete" rather than 0. This broke the ordinary "replay" use case for *any* top-level `six.to()`/`six.timeline()` once played to completion, found while building a Timeline feature-coverage demo (`public/timeline-demo.html`) where repeated Play-button clicks stopped animating after the first. Also revealed the same gap applied to `.reverse()`: changing `timeScale()`'s sign on a live (non-paused) child never re-anchored `_start` either, so reversing something that had been running for a while didn't animate backward at all - it clamped straight to 0 on the very next frame. And `play()`/`resume()` had been collapsed into one identical method (`{ this.paused(false); }`), so there was no way to force forward playback after a reverse - contradicts GSAP's actual, distinct semantics (`play()` always plays forward; `resume()` and `reverse()` preserve whatever direction is set). Fixed in `core/animation.ts`: `timeScale()` now calls `this.parent?._childResumed?.(this)` unconditionally on every change (reused by `restart()` via `play()`, and by `.reverse()`/`.timeScale()` directly), `play()` now calls `reversed(false)` before unpausing, and `resume()`/`reverse()` call `paused(false)` directly instead of delegating to `play()`. Covered by new tests in `animation.test.ts` and `timeline.test.ts`.
5. **A child paused inside a *bounded* nested timeline could end up permanently frozen on resume if enough real time passed first, even though the exact same pause/resume worked fine when done quickly.** `_childResumed`'s re-anchor math (bug 1) used `this._lastRenderedLocal` - the parent timeline's own last-rendered local time - as "now." That cached value goes stale the moment the parent itself stops being actively rendered, which is exactly what happens to a bounded nested timeline once its own span is exceeded: `Timeline._renderIteration`'s skip-range optimization (`if (end < lo || start > hi) continue`) means its *own* parent stops calling `render()` on it entirely once real time passes its end, even while it's still logically "not paused." Once that happens, `_lastRenderedLocal` freezes; a `_childResumed` reanchor computed off it afterward understates how much real time has actually passed, producing a startTime that still falls short of "now" - which usually means the child's own newly-extended span *still* doesn't reach the current time, so the parent keeps being skipped forever and the child is stuck at whatever position it was paused at, permanently. Confirmed both broken and fixed by actually running reproductions through the real public API (`six.timeline()` + `rootTimeline`) via `vitest`, not by inspection alone - a stale-value trace like this is easy to get wrong by hand. Fixed with a `currentLocalTimeOf()` helper (`timeline/timeline.ts`) that computes a timeline's true current local time by walking up through parents via each ancestor's own current `totalTime()` (bottoming out at the ultimate root, which is never skipped and so is always accurate) instead of trusting any single cached `_lastRenderedLocal`. `_childResumed` now uses this instead of `this._lastRenderedLocal`. Covered by a new test in `timeline.test.ts`.
6. **A zero-duration animation (`.set()`/`.call()`) added "now" onto the always-ticking root could get a redundant second render one real tick after creation, silently reverting any *later*-created sibling's write to the same target back to its own value.** A `Tween`'s constructor already self-renders once synchronously (`this.render(0, true, true)`), applying a `.set()`'s values immediately - that's correct and sufficient, since a zero-duration animation's output is `eased(1)` regardless of *when* it's rendered (never a function of elapsed time). But because the synchronous script that creates it never advances root's own clock, the child's `startTime` (root's "now" at creation) ends up exactly equal to root's `_lastRenderedLocal` from the *previous* real tick - and `Timeline._renderIteration`'s boundary-inclusive skip check (`end < lo`, not `<=`) treats that as "not yet rendered," giving it one more real render through the parent on the very next tick. That second render is completely redundant on its own - but if a sibling created *after* it in the same script (e.g. a `six.to(..., { scrollTrigger })` reveal that gets synchronously jumped to its complete state - see the `ScrollTrigger` "already past on construction" fix above) already wrote a *different* value to the same target before that tick fires, the zero-duration child's redundant render silently stomps it back. Found while debugging a report that a scroll-reveal's first few words never appeared - traced through real browser console output (`getComputedStyle` showing the pre-set random `y`/`rotate` values, not the revealed ones, even though the tween's own `totalProgress()` correctly read `1`) down to a `ticker.tick()`-driven reproduction, not by inspection alone. Fixed in `Timeline._renderIteration`: additionally skip a child when `child.totalDuration() === 0 && start <= lo` - a zero-duration child only still needs its one real render when its scheduled position is *strictly ahead* of the low-water-mark (genuinely not yet reached), not when it merely coincides with it. Covered by two new tests in `timeline.test.ts` (one confirming the redundant render is now skipped, one confirming a genuinely future-scheduled zero-duration child still renders once it's actually reached).

**`.reverse()`ing an infinite-repeat (`repeat: -1`) animation that had ever been paused/resumed (or otherwise re-anchored) permanently corrupted it to `startTime = -Infinity`, breaking every future render and making it unrecoverable even via `.restart()`/`.play()`.** Found live via the `svgMotion` demo (`svg-motion-demo.html`): a `repeat: -1`, `autoRotate`-following plane on a `motionPath` had a "Reverse" button; clicking it once and the plane never moved sanely again, not even after "Restart". Root cause: `Timeline`'s `toChildTotalTime()`/`_childResumed()` (`timeline.ts`) both compute a `reversed`-playback offset as `ts >= 0 ? 0 : tDur` - the idea being that reversed playback starts counting down from the animation's own end (`tDur`) rather than up from 0. For `repeat: -1`, `totalDuration()` is `Infinity` (see `cycle.ts`'s `totalDurationOf`) - there is no "end" to count down from, so using `tDur` directly there feeds `Infinity` straight into the surrounding arithmetic, and `_childResumed`'s re-anchor (`now - (childTotalTime - offset) / ts`) collapses to `-Infinity`. Every later render for that child then computes NaN/nonsense localTime. Fixed with a small shared `reverseOffset(ts, tDur)` helper (replacing both inline `ts >= 0 ? 0 : tDur` expressions) that falls back to `0` whenever `tDur` isn't finite - structurally identical to the forward-playback case, i.e. "keep counting from wherever totalTime already is" - which is the only coherent reading of "reversed" when there's no finite end to reference. Verified broken and fixed via a direct repro (`new Tween(el, { repeat: -1 }).reverse()` under a live `Timeline`) before writing the regression test in `timeline.test.ts`. This is a general core bug, not specific to `svgMotion` - any `six.to()`/`six.timeline()` with `repeat: -1` that gets `.reverse()`d after having been paused/resumed once (which includes anything driven by `OnScroll`, since `vars.animation.pause()` runs at construction) would have hit it.

**`TweenVars.overwrite?: boolean | "auto"` is now implemented** (`tween/overwrite.ts`): a `WeakMap<Element, Map<string, Tween>>` tracks which `Tween` currently "owns" each `(target, prop)` pair. Every tween registers its own claimed pairs on init, regardless of its own `overwrite` setting (so a *later* tween can detect and overwrite it even if *this* one didn't ask for protection) - `overwrite` unset/`false` still results in no kills, matching GSAP's own default. `overwrite: true` kills the *entire* previous owning tween (all its targets/properties, not just the conflicting one) - matches GSAP's documented "regardless of which properties are being animated" behavior. `overwrite: "auto"` instead surgically removes just the conflicting `(target, prop)` track from the owner via `Tween._dropTrack()`, leaving its other properties/targets animating undisturbed, and only kills the owner outright if that was its last remaining track. Required `ColorTrack`/`ComplexTrack`/`DiscreteTrack` to start carrying their `prop` name too (previously only `NumericTrack` did) so `"auto"` can match on it. Keyframe-based tweens are excluded (documented scope cut - `_onInit` only calls `applyOverwrite` in the non-keyframe branch, since a keyframe tween's tracks are built lazily per-segment rather than up front). Covered by tests in `tween/overwrite.test.ts`.

**Only Core, Tween, Timeline, Animate, and ScrollTrigger are implemented.** SplitText, Draggable, Flip, Observer (the gesture primitive, not `scroll-trigger/observer.ts`), ScrollSmoother, and MotionPath are explicitly out of scope for this phase. Property renaming, API trimming, six-js-specific features, and performance tuning are also explicitly deferred until Phase 1 is complete and reviewed — don't do any of that opportunistically while touching this code.

**`LEGACY_CLEANUP.md`** lists old-prototype files (`src/components/`, `src/plugins/`, `src/properties/`, several `src/core/*.ts`) that are dead but still physically present — a bulk-delete was blocked by a safety gate during the autonomous rewrite session. They're excluded from `tsconfig.json` and unreferenced from `src/index.ts`, so they don't affect builds/tests, but they should be deleted (and the tsconfig excludes removed) the next time someone can authorize it.

## Architecture

### Core (`src/core/`)

- **`ticker.ts`** — `Ticker` class, one shared instance (`ticker`) drives everything. rAF-based with lag smoothing (a backgrounded tab doesn't cause a catch-up jump on refocus) and a `{ manual: true }` mode (bypasses rAF/the overlap-gap gate entirely, driven only by `.tick(deltaMs)`) used throughout the test suite for deterministic timing.
- **`animation.ts`** — `Animation`, the abstract base both `Tween` and `Timeline` extend. Owns the entire time model (`_start/_dur/_tDur/_time/_tTime/_ts/_rts`) and lifecycle (`play/pause/resume/reverse/restart/seek/kill/invalidate`, event emitter). `_ts === 0` *is* the definition of paused; the sign of `_rts` *is* the definition of reversed — no separate boolean flags to keep in sync. `render(totalTime, suppressEvents, force)` is concrete (not abstract): it does the generic clamping/repeat-cycle/event-firing work and delegates only the "apply visual state for this local time" part to the abstract `_renderIteration()`. **Deliberately does NOT skip re-rendering when `totalTime` looks unchanged** — see the long comment in `render()`; two different logical render passes (e.g. a sibling touching the same target in between) can compute the identical totalTime for an animation, and skipping on value-equality alone silently drops a needed re-render. A zero-total-duration animation (`.set()`/`.call()`) always clamps to totalTime 0, so before/after boundary detection there uses the *raw* pre-clamp totalTime's sign (`_rawPrev` field) instead of the clamped value. `paused(false)` (resuming) calls `parent?._childResumed?.(this)` — see `Timeline._childResumed` — a real, high-impact bug (animations jumping to their end state on resume) lived here; read the Status section above before touching pause/resume. `restart(includeDelay)` totalTime target is `includeDelay ? 0 : this._delay` — get this backwards and the two modes become indistinguishable (both clamp to the same value), which is exactly the bug that shipped once already.
- **`cycle.ts`** — `resolveCycle()`, the one shared repeat/yoyo formula (floor-division based) both `Animation.render()` call sites use, instead of GSAP's two independently-duplicated inline copies.
- **`linked-list.ts`** — generic sorted doubly-linked-list helpers (`insertSorted`/`removeNode`/`forwards`/`backwards`), reused for `Timeline` children.
- **`root.ts`** — `rootTimeline`, the one `Timeline` registered with `ticker`. Constructed `{ unbounded: true, defaultPosition: "now" }`: `unbounded` means its own `totalDuration()` is always `Infinity` (never clamped by whatever children currently exist — without this, once ticker time exceeds the current max child span the root's own `_tTime` gets stuck and can never advance again even as new children are added), and `defaultPosition: "now"` means a child added with no explicit position starts at the root's *current* playhead, which is what makes independently-created top-level tweens play immediately/concurrently instead of queuing. Kept always-on for Phase 1 rather than implementing GSAP's autoSleep (rAF-loop teardown when idle) — documented scope cut, not an oversight.
- **`context.ts`** — `Context`/`context()` (`six.context()`). A module-level active-scope stack; `Animation`'s constructor calls `getActiveScope()?._capture(this)` unconditionally, so both `Tween` and `Timeline` get auto-capture-on-create for free without `context.ts` knowing either class exists (it only deals with a generic `Killable` interface — zero dependency direction from context → animation).
- **`defaults.ts`** — `getDefaults()`/`setDefaults()` (`six.config()`), just `{ duration, ease }` fallbacks.

### Tween (`src/tween/`)

- **`tween.ts`** — `Tween extends Animation`. Normally builds `PropertyTrack[]` from vars via `animate/registry` lazily on first render (`_onInit()`); when `vars.keyframes` is given it instead composes (has-a, not is-a) an internal `Timeline` of `.fromTo()` segments (`keyframes.ts`) and just drives that from `_renderIteration()`. The constructor always calls `this.render(0, true, true)` immediately so e.g. a `.from()` tween's starting values are visible synchronously, not one tick later. `_onInit()` calls `applyOverwrite()` (`overwrite.ts`) right after building tracks, for non-keyframe tweens only. `_dropTrack(target, prop)` (called by `overwrite.ts`'s `"auto"` mode) surgically removes one track and self-kills once none remain.
- **`overwrite.ts`** — implements `TweenVars.overwrite`. A `WeakMap<Element, Map<string, Tween>>` tracks which `Tween` currently owns each `(target, prop)` pair; every tween registers its own claimed pairs on init regardless of its *own* `overwrite` setting (so a later tween can still detect and overwrite it, even though *this* one didn't request protection) — leaving `overwrite` unset costs nothing extra and correctly kills nothing, matching GSAP's own default (`false`). `overwrite: true` kills the *entire* previous owner (all its targets/properties, not just the conflicting one) — matches GSAP's documented "regardless of which properties are being animated" behavior for `true`. `overwrite: "auto"` instead calls the owner's `_dropTrack()` for just the conflicting `(target, prop)` pair, leaving its other properties/targets undisturbed, and only kills it outright once it has no tracks left.
- **`property-track.ts`** — the actual interpolation records (`NumericTrack | ColorTrack | ComplexTrack | DiscreteTrack`, all four now carrying their own `prop` name so `overwrite.ts` can match on it) and `buildTracks()`/`applyTrack()`. Transform properties write into the shared per-element cache (`animate/transform-cache.ts`); `Tween._renderIteration()` collects which targets were touched and composes+writes the final `transform` string **once per element per frame**, not once per transform sub-property.
- **`keyframes.ts`** — `buildKeyframeTimeline()`. Each keyframe becomes a `.fromTo()` segment; a `carry` object explicitly threads each segment's own end values forward as the next segment's start (rather than letting the next segment "read current DOM state", which by the time its own track is built lazily may not yet reflect the previous segment's outcome).

### Timeline (`src/timeline/`)

- **`timeline.ts`** — `Timeline extends Animation`, implements `AnimationParent` (children call `parent._removeChild(child)`/`parent._uncache()` — structural interface, so `core/animation.ts` never imports `Timeline`) and `ListHandle<Animation>` (backs the generic linked-list). `_recomputeTotalDuration()` scans children for `max(child.endTime())`. `_renderIteration()` converts parent-local time into each child's own totalTime via `toChildTotalTime()` (`(parentLocalTime - child.startTime()) * ts + (ts >= 0 ? 0 : tDur)`) — implemented purely through `Animation`'s public API (`startTime()/timeScale()/paused()/totalDuration()`), never protected-field access, so there's no coupling between sibling subclasses. **Only renders children whose `[start, end]` overlaps the span actually crossed since the last render** (`lastLocal..localTime`, in either direction) — rendering every child unconditionally every frame was tried first and is unsafe whenever two children share a render target with different values (chained keyframe segments, or two `.to()` calls on the same property at different timeline positions): a still-in-the-future child clamps to *its own* start value and, iterating after an active sibling in start-time order, silently overwrites the active one's just-computed value. `Timeline.add()` widens the low-water mark back to a newly-added child's start so a child inserted behind the current playhead is still picked up on the next render. A child with `totalDuration() === 0` (a `.set()`/`.call()`) is additionally skipped once `start <= lo` - see bug 7 in the Status section for why a zero-duration child's render is provably idempotent once its position has already been reached, making a second one not just wasteful but actively harmful if a sibling wrote to the same target in between. `kill()` is overridden to cascade to every child before detaching itself (the base `Animation.kill()` only detaches from its own parent). `_childResumed(child)` implements the `AnimationParent` hook `Animation.paused()` calls on resume (and that `timeScale()` now also calls on every change, and `restart()` via `play()`) — re-anchors `child.startTime()` using `currentLocalTimeOf(this)` (this timeline's own *true current* local time, computed fresh by walking up through parents rather than trusting the potentially-stale cached `_lastRenderedLocal` — see bug 5 in the Status section) so the child continues from its frozen totalTime instead of jumping to wherever the unadjusted `(parentLocalTime - _start)` formula now points. Guarded by `now < child.startTime()` (not by whether the child's totalTime had already progressed) — see the long comment on the method for why the naive "hasn't progressed yet" check gets the most common real case (a ScrollTrigger-driven, `paused: true`-from-creation animation resumed later) backwards.
- **`position-parser.ts`** — pure `resolvePosition()` for `"<"`/`">"`/`"+=1"`/label-reference position strings, independently testable against a small context object (no `Timeline` construction needed).
- **`stagger.ts`** — `computeStaggerDelay()`. Not a parallel "stagger group" primitive: `Timeline.to(targets, { stagger })` builds one `Tween` per target, each with its own `delay`, all added at the *same* timeline position — reuses `Timeline`/`Tween` entirely.

### Animate (`src/animate/`)

The property/DOM-rendering layer — what `Tween` calls to actually read/write the DOM (GSAP's `CSSPlugin` equivalent).

- **`registry.ts`** — `resolveHandler(target, prop, sampleValue)` dispatches to a `numeric | color | complex | discrete` handler. Resolution ladder: transform alias → known color/complex prop → CSS custom property (`--foo`, numeric vs discrete decided from `sampleValue`) → generic style property (numeric if `sampleValue` looks numeric) → genuine plain-number non-style property (`scrollTop`-style; **only** when `typeof target[prop] === "number"`, not merely `prop in target` — an existing-but-non-primitive property like an SVG element's `cx` (`SVGAnimatedLength`) must fall through to `setAttribute` instead of being naively overwritten with a raw number) → `setAttribute` fallback.
- **`transform-cache.ts`** — per-element cache (`WeakMap<Element, TransformCache>`, not an expando property — a deliberate simplification vs GSAP's `target._gsap`, revisit only if profiling says it matters). `parseMatrixString()` (exported standalone, unit-testable without a real browser) decomposes an existing `matrix()` fully (translate/rotate/scale/skew) but only reads translate x/y/z out of an existing `matrix3d()` — full 3D decomposition is a documented Phase 1 scope cut. `composeTransform(cache, use3D)` prefers `translate3d` while `use3D` is true (the caller passes `ratio > 0 && ratio < 1`, i.e. actively animating), falling back to plain `translate()` at rest so elements aren't left on a GPU compositor layer unnecessarily — **but a nonzero `z` always forces `translate3d` regardless of `use3D`**. It used to fall back to `translate(x, y)` purely off `use3D`, which silently dropped `z` entirely (that function only takes 2 args) the instant any tween touching `z` came to rest - a real, verified bug, not a hypothetical one; `use3D` is a compositing-performance hint for the default `z === 0` case, never a license to lose real data.
- **`unit-convert.ts`** — measurement-based unit conversion (rem via root font-size, em via the target's own font-size, vh/vw via viewport size, % via the parent's `clientWidth`/`clientHeight`). The % path needs real layout and so only produces a meaningful result in a browser, not jsdom (documented in the file and in the test suite).
- **`color.ts`** — parses hex/rgb/rgba (comma and modern space syntax)/named colors. Named-color resolution is a **hardcoded table**, not resolved through the DOM/CSSOM (`element.style.color = name; read back`) — jsdom's CSSOM doesn't normalize named colors the way real browsers do, which would make parsing silently environment-dependent; the hardcoded table is deterministic everywhere.
- **`complex-string.ts`** — tokenizes numbers *and* embedded colors out of multi-value CSS strings (`box-shadow`, etc.), interpolates each positionally, reassembles using the end string's literal segments as the template. `canInterpolateComplex()` gates whether `property-track.ts` uses this path at all vs degrading to a discrete swap when start/end have structurally different token shapes.

### ScrollTrigger (`src/scroll-trigger/`)

Split to mirror GSAP's own `Observer.js`/`ScrollTrigger.js` separation.

- **`observer.ts`** — generic scroll/resize listening + a memoized-read layer keyed off a module-level "generation" counter (`invalidateReads()` bumps it; `getScroll()` only performs the real, layout-touching read once per generation no matter how many callers ask). Zero dependency on `scroll-trigger.ts`. `addResizeListener()` also re-runs listeners once on `window`'s `load` event if `document.readyState` isn't `"complete"` yet when first called - a `ScrollTrigger`'s first `refresh()` (during construction) can run before images/fonts below the trigger finish loading, so without this its initial start/end measurement is taken against a document that's still going to grow.
- **`pin.ts`** — spacer creation is cached **per pinned element** (`WeakMap<HTMLElement, ...>`, reference-counted), not per `ScrollTrigger` instance, so multiple triggers pinning the same element share one spacer. Three explicit phases (`"before" | "during" | "after"`) rather than toggling a single `position: fixed` on/off: `"after"` uses `position: absolute` with `top: distance` *within the spacer* so the element resumes normal document flow once scrolled past, instead of staying artificially fixed. The `"during"` phase's `top` is **not** hardcoded to `0` - it's `naturalDocTop - startY` (`setPinnedTop()`, computed in `scroll-trigger.ts`'s `refresh()`), i.e. wherever the element naturally sat in the viewport at the moment scroll reaches `startY`. Hardcoding `top: 0` was a real shipped bug: it only looked correct for a `start: "top top"` trigger (where that offset genuinely is 0) and silently snapped anything pinned via `"center center"` or similar to the viewport's top edge instead of its natural (e.g. vertically centered) position - and made the element flicker between that wrong position and its normal unpinned one whenever scroll hovered right at the `start` boundary, since `"before"`/`"during"` disagreed about where it should be. `distance` (`endY - startY`) is what keeps the `"during"`→`"after"` transition seamless regardless of `pinnedTop`'s value - this falls out of the geometry algebraically, not from any extra `"after"`-phase adjustment.
- **`scrub.ts`** — `createDirectScrub()` sets progress synchronously 1:1. `createSmoothScrub(animation, seconds)` retargets on each scroll event: fetched and verified against GSAP's actual `ScrollTrigger.js` source (`scrubDuration`) rather than inventing a plausible-looking formula - real GSAP builds an internal tween with `ease: "expo"` (i.e. `expo.out`) and `duration: scrub`, then on every scroll event either calls `scrubTween.resetTo("totalProgress", newTarget, ...)` or (fallback) sets `vars.totalProgress` and calls `.invalidate().restart()` - either way, a **fresh `expo.out` ease from the current value to the new target, over exactly `scrub` seconds**, restarted from elapsed-zero on every retarget. Our first implementation instead used a continuous, never-restarting exponential-decay loop (`current += (target-current)*(1-e^(-3dt/scrub))`) - shaped similarly (fast-then-slow) but with a much weaker initial response than `expo.out` (initial rate `3/duration` vs `expo.out`'s `~6.93/duration`), so for the same declared `scrub` number it visibly lagged behind real GSAP, most noticeably right after a single scroll tick rather than at full settle. Fixed to match: `from`/`to`/`elapsed` state, `ease(elapsed/duration)` each tick, `elapsed` reset to 0 whenever `update()` retargets - drives `animation.totalProgress()` directly rather than via a helper `Tween`, since `Tween` is DOM-property-oriented and this is a single plain numeric value. Exposes **two** entry points, not one: `update()` (scroll-driven, smoothed once *settled*) and `snapTo()` (recalculation-driven, always instant). `update()` keeps snapping instantly - not smoothing - until this scrub's own ticker tick has fired at least once since creation. This matters for a real bug: reloading mid-page, the browser's own async scroll-position restoration doesn't always land before this controller's first read of `window.scrollY`, so the very first read can be stale (e.g. 0), and the browser's restoration then fires its own native `scroll` event shortly after with the real position - if that were treated as an ordinary smoothed update, the tween would visibly "rewind" to the stale position and ease back into place on every reload, even though nothing the user did should have caused any animation. Read the long comment at the top of `createSmoothScrub` before changing this - two mitigations are needed together (the `settled` gate here, plus `scroll-trigger.ts`'s `refresh()` always calling `snapTo()`) because neither alone is airtight given browser restoration timing isn't specified/guaranteed.
- **`scroll-trigger.ts`** — `ScrollTrigger` class + `parseEdge()`/`resolvePositionString()` (pure, exported, unit-tested independent of any DOM measurement) for `"<edge> <edge>"` position strings (`top|center|bottom|N%|Npx`, default `start: "top bottom"` / `end: "bottom top"`). `resolvePositionValue()` special-cases a pure `"+=N"`/`"-=N"` string (no edge tokens) as **N pixels relative to `end`'s `relativeBase` parameter** (the already-resolved `startY`) rather than routing it through the edge parser — this is the standard `end: "+=500"` idiom for expressing a pin/scrub distance directly in pixels regardless of the trigger's own height; **do not remove this branch** - without it, `"+=N"` falls through to `parseEdge()`, where `parseFloat("+=N")` is `NaN` (the `+`/`=` characters aren't valid for a plain number), silently resolving to `{ratio:0, offsetPx:0}` and collapsing `end` to essentially the same point as `start`. The visible symptom of that regression is dramatic, not subtle: the entire scrubbed animation compresses into a ~1px scroll range, so it appears to fire instantly and the smoothed-scrub progress oscillates wildly between 0 and 1 from ordinary scroll jitter (looks exactly like a jittery/broken animation, not an off-by-a-little position bug) - this is exactly the bug a user hit with the classic `end: "+=${distance}"` horizontal-scroll-gallery pattern. `refresh()` reverts any pin to baseline *before* measuring (so a currently-pinned element's transformed state can't corrupt its own measurement), resolves start/end and pin distance, then calls `update(true)` (`instant`) so a recalculation always repositions synchronously and never animates as a side effect - see `scrub.ts`. `update()` computes progress and fires `onEnter/onLeave/onEnterBack/onLeaveBack` based on scroll direction. When not scrubbing, the default (and *only* supported - no `toggleActions` string configurability) behavior matches GSAP's own default `toggleActions: "play none none none"`: **only** crossing into range going forward calls `.play()`; entering backward, leaving forward, and leaving backward do nothing to the animation by default (their callbacks still fire). This was a real bug too, in the other direction from most of the others: it initially called `.reverse()` on leaveBack, which is a plausible-sounding but wrong default - it makes a scroll-reveal animation silently undo itself every time the user scrolls back up past the trigger, which is not what "no toggleActions specified" means in GSAP. `onUpdate` only fires while `inside` the trigger's range, or on the exact frame of entering/leaving it - **not** unconditionally on every scroll event site-wide. Progress is clamped (constant) outside the range, so a scroll anywhere else on the page produces no actual change for this instance; firing onUpdate for those was a real reported bug (it fired continuously while scrolling through unrelated page content, long before/after the trigger's own range). `pin` is validated as `true`/a selector/an `Element` right after resolution - anything else (e.g. a stray number) logs a warning and disables pinning for that instance instead of crashing inside `pin.ts` with a confusing `getBoundingClientRect is not a function`. No batched cross-instance refresh ordering, no `anticipatePin`, no snap, no custom-scroller-proxy system beyond a plain `Element` or `window`, no `start: "+=N"` support (only `end`, since `start` has no prior resolved value to be relative to) — documented Phase 1 scope cuts.

**A trigger constructed already behind the current scroll position (e.g. a page reload deep into a long scroll-reveal sequence) used to visibly replay its animation from scratch instead of showing it already complete.** `update()`'s default-toggle branch (`inside && !wasInside` → `.play()`) doesn't distinguish "genuinely just crossed during live scrolling" from "was already past this point before we ever started watching" - a fresh `paused: true` tween is still frozen at totalTime 0 regardless of which case applies, so calling `.play()` on it always animates from the beginning. On a page reloaded mid-scroll, every trigger whose `start` already lies behind the restored scroll position would `.play()` at construction and visibly animate in, all at once, looking like the whole page's reveals are replaying on load. Worse: a trigger already past its *own* `end` too (common a few triggers back on a long page) never got `.play()`d at all, since `inside` requires `scrollY <= endY` - `onEnter` never fires, so it stays invisible forever. Fixed with a `hasEverUpdated` flag: on a trigger's first-ever `update()` call, if `scrollY >= startY` already (regardless of `endY`), `onEnter` fires but the animation jumps straight to `totalProgress(1)` instead of `.play()`ing - no visible replay, and it's correctly shown complete even when already past its own end too. Only applies to non-scrub triggers (`!scrubController`) - a scrub trigger already self-corrects via `refresh()`'s `scrubController.snapTo()`. A genuine live-scroll crossing *after* construction is unaffected (`hasEverUpdated` is already `true` by then), so the normal reveal animation still plays as expected. Covered by two new tests in `scroll-trigger.test.ts`.

**`horizontal`/`containerAnimation` (GSAP's pattern for reveals nested inside a pinned, horizontally-scrubbed track) are now implemented.** `horizontal: true` measures/compares along `left`/`right`/width instead of `top`/`bottom`/height - `EDGE_KEYWORDS` gained `left:0, right:1` alongside the existing `top:0, bottom:1` (same ratios, so `parseEdge`/`resolvePositionString` needed zero changes; only the caller's rect/axis construction is axis-aware, via `axis()`/`axisRect()`). `containerAnimation: someAnimation` ties this trigger's notion of "current scroll position" to `someAnimation.totalProgress()` instead of native scroll, and updates are driven by subscribing to the container's own `"update"` event (`Animation.on`) rather than `addScrollListener` - a scrubbed container re-renders every ticker frame via its own smoothing, not just on native `scroll` events, so listening to *its* updates is what actually keeps a nested trigger in sync. `start`/`end` position strings for a `containerAnimation` trigger are resolved against the trigger element's **measured position at the container's totalProgress 0 and 1** (`measureContainerEdges()`, via silent `seek()` calls so no stray `onUpdate`/events fire during the probe) rather than real scrollY/scrollX, producing a **progress ratio** (not a pixel value) stored in the same `startY`/`endY` fields - this assumes the container's own motion is linear (`ease: "none"`), which is the standard convention for a scrubbed container (verified against GSAP's real ScrollTrigger.js source: it also assumes/requires this, converting ratios through the container's own linear time-vs-scroll relationship). A relative `end: "+=N"` for a `containerAnimation` trigger is deliberately reinterpreted as **N pixels of the trigger element's own measured travel** (`|edge1 - edge0|`), converted to an equivalent progress delta - simpler and more intuitive than GSAP's actual internal unit (seconds along the container tween's own duration axis, an implementation detail its real source exposes but that we don't try to replicate byte-for-byte). Debug markers are not supported in `containerAnimation` mode (not exercised by anything using this feature yet - a scope cut, not an oversight). Covered by `scroll-trigger.test.ts`'s "containerAnimation/horizontal" suite.
- **`markers.ts`** — debug-only start/end lines (`markers: true`), isolated so it stays out of bundles that never enable it. Uses `position: absolute` at the *document*-Y coordinate (not viewport-relative), so it naturally scrolls with the page and only needs updating on refresh, not on every scroll event. Labels render **below** their line (not above via a negative `translateY`) - a marker positioned near the very top of the document would otherwise have its label pushed off-screen above `y: 0`. `vars.id` (`ScrollTriggerVars.id`) shows up in the label text if provided, else a bare instance number, matching GSAP's own markers.

### SvgMotion (`src/plugins/svg-motion/`)

Three SVG effects bundled as one plugin (`six.svgMotion.draw/morph/motionPath`) — GSAP's DrawSVGPlugin + MorphSVGPlugin + MotionPathPlugin combined, per the actual ask rather than three separate plugins. Architecturally different from every other plugin in this codebase: `DrawAnimation`/`MorphAnimation`/`MotionPathAnimation` each **subclass `core/animation.ts`'s `Animation` directly** (exactly what `Tween` and `Timeline` themselves do), instead of going through the `PropertyTrack` system (`tween/property-track.ts`) or driving the ticker directly (`burst.ts`'s approach). None of the three fit a `PropertyTrack` (dasharray needs the path's total length up front; morph needs pre-sampled point sets; motionPath needs a live path query every render) - subclassing `Animation` gets play/pause/resume/reverse/restart/repeat/boomerang/delay/kill/context-capture/events/Timeline-nesting for free without adding a fourth track kind to core, and without modifying `property-track.ts`/`registry.ts`/`tween.ts` at all (every other plugin here follows the same "don't touch Tween/Timeline internals" rule).

- **`svg-geometry.ts`** — `resolveGeometry()` resolves an `Element | string` shape input: an existing element is used as-is, a CSS selector that resolves to a geometry-capable tag uses that match, anything else (including a syntactically invalid selector, e.g. raw `"M0,0 L10,10"` path data) is materialized into a **detached** `<path>` (never appended to the DOM - per spec, `getTotalLength`/`getPointAtLength` work on detached geometry too) - this is what lets `motionPath`'s `path` and `morphSVG`'s `toShape` accept a raw `d` string with no visible/rendered guide element required, matching GSAP's own MotionPathPlugin. Tag-name-based (`isGeometryElement`), deliberately **not** `instanceof SVGGeometryElement` - that global doesn't exist at all in jsdom (the test environment), so an `instanceof` check would throw a `ReferenceError` the moment a test imports the module, not just fail the check. `shapeToPath()` converts `circle`/`ellipse`/`rect`(rounded or not)/`line`/`polyline`/`polygon` into an equivalent `d` string (a `path` is returned as its own `d` unchanged) so `drawSVG`/`morphSVG` can operate on any basic shape, not just literal `<path>` elements.
- **`draw.ts`** — `DrawAnimation` animates `stroke-dasharray`/`stroke-dashoffset` to reveal (or hide) a shape's stroke - the classic "line draw" effect, e.g. drawing in a signature or a word made of letter outlines. `from`/`to` are each a ratio or an explicit `[start, end]` window (a single value means `[0, value]`, the ordinary fixed-start reveal; a two-token `"20% 80%"` moves both ends independently, for a moving "comet" segment). `drawSVG(target, vars)` resolves `target` like any other six-js target - resolving to more than one shape (e.g. `drawSVG("svg .letter", { stagger: 0.15 })` for a whole word) builds one `DrawAnimation` per shape and groups them in a `Timeline`, same "reuse `Timeline`, don't invent a group type" principle `six.to()`'s own stagger support uses.
- **`motion-path.ts`** — `MotionPathAnimation` moves a target along a guide path (e.g. a plane following a wavy SVG curve) by writing into the **same shared per-element transform cache** every ordinary tween writes into (`getTransformCache`/`renderTransform` from `animate/transform-cache.ts`), not a competing `element.style.transform =` write of its own - so it composes cleanly with any other transform tween running on the same element instead of one clobbering the other. Applied as a **delta from the path's own start point** (matching GSAP's real MotionPathPlugin behavior): the target keeps whatever base x/y it already had, and moves by the same amount the path point itself moves relative to progress 0 - it does not jump to the path's absolute SVG coordinates, which are usually a totally different coordinate space than the target's own page position. `autoRotate: true | number` faces the target along the path's tangent (via a small symmetric before/after sample, not a forward-only difference, so the first/last frame still gets a meaningful direction); a number adds a fixed degree offset on top, to correct for whichever way the target's own artwork points by default.
- **`morph.ts`** — `MorphAnimation` animates `target`'s own `d` attribute from its current shape into `toShape`'s (GSAP's MorphSVGPlugin) by sampling both shapes into equal-length point sets (`samplePoints()`, via `getPointAtLength`) and interpolating each pair positionally, reconstructing a smooth curve each frame via a uniform Catmull-Rom → cubic-bezier conversion (`pointsToBezierPath()`) rather than a faceted polyline. **Not** GSAP's own shape-fitting/point-reduction algorithm - a documented scope cut; good enough for typical blob/icon/letterform morphs, but very different topologies may show simple straight-line-interpolation artifacts. `alignToFrom()` picks whichever rotation/winding-direction of the sampled `to` points best lines up index-for-index with `from` (minimizing total squared point distance) before interpolating - both shapes are sampled independently starting from their own path's start, which for two unrelated shapes usually land at very different physical spots/windings, so interpolating them unmodified visibly "twists" the morph (a real, easy-to-see artifact). Only tries rotations for **closed** shapes (open paths only try forward vs reversed - there's no "wrap around" on an open path). `samplePoints`/`alignToFrom`/`lerpPoints`/`pointsToBezierPath`/`isClosed` are all exported standalone (like `scroll-trigger.ts`'s `parseEdge`) so the point math is unit-testable against mocked geometry, independent of the class.

**Testing note**: jsdom implements **no SVG geometry methods at all** - `getTotalLength`/`getPointAtLength` don't exist on any SVG element regardless of tag or DOM attachment (unlike the layout-method gap documented below, this isn't even attachment-dependent). `vi.spyOn` requires the method to already exist, so tests assign it directly (`(el as unknown as {...}).getTotalLength = () => length`) rather than spying. For a raw `d`-string path input (materialized internally by `resolveGeometry`, so there's no element reference to pre-mock before construction), the only option is a shared `SVGElement.prototype` mock, cleaned up in a `finally` block - see `motion-path.test.ts`'s "accepts a raw path `d` string" test.

### Public API (`src/api/six.ts`)

`six.to/from/fromTo/set/timeline/config/context`, plus `ScrollTrigger` re-exported. `six.to()` etc. attach the created `Tween`/`Timeline` to `rootTimeline`; with `stagger`, wraps the per-target tweens in a `Timeline` (returned to the caller) rather than a bare array, so the whole staggered group can still be paused/reversed/killed as one unit — same "reuse `Timeline`, don't invent a group type" principle Timeline's own stagger support uses. `scrollTrigger` in `TweenVars`/`TimelineVars` auto-creates a `ScrollTrigger` wired to the (paused) animation; `trigger` defaults to the tween's own target when omitted (a `Timeline` has no inherent target, so `scrollTrigger.trigger` is required there). `six.svgMotion = { draw, morph, motionPath }` (see the SvgMotion section above) - each factory attaches its own `Animation` subclass straight to `rootTimeline` the same way `six.to()` does.

## Testing

`vitest` + jsdom. `Ticker` tests always construct `new Ticker({ manual: true })` (never the shared `ticker` singleton) to avoid real rAF timing. Tests against `rootTimeline`/`ticker` (in `api/six.test.ts`) capture a starting point (`rootTimeline.totalTime()`) and only assert relative to it, since `rootTimeline` is a real module-level singleton shared across every test in the process — asserting against absolute values makes tests order-dependent.

**Mock `window.scrollY`/`window.innerHeight` via `vi.spyOn(window, "scrollY", "get")`, never `Object.defineProperty`** — `vi.restoreAllMocks()` in `afterEach` does not undo a raw `Object.defineProperty` override, so it silently leaks into later tests in the same file (looks exactly like a real caching bug when it's actually test-isolation; this cost real debugging time once already — see `scroll-trigger.test.ts`/`six.test.ts`). Similarly, when manually simulating a scroll event that should invalidate `observer.ts`'s read cache, call `invalidateReads()` directly rather than relying on the dispatched event to trigger it — there's no listener registered to catch the event until a `ScrollTrigger` has actually been constructed, so `scrollTo()`-before-construction (setting up the initial position) would otherwise silently no-op.

jsdom does not implement layout: `getBoundingClientRect()` returns all-zero rects and `clientWidth`/`clientHeight`/`scrollHeight` are always 0 unless a test explicitly mocks them (`vi.spyOn(el, "getBoundingClientRect").mockReturnValue(...)`, as `scroll-trigger.test.ts` and `pin.test.ts` do). The `%` unit-conversion path in `animate/unit-convert.ts` is untestable in jsdom for this reason and is only exercised by the browser smoke test (`index.html`).
